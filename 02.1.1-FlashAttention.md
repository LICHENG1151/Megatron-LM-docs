# 02.1.1 · FlashAttention（把注意力激活从 O(s²) 压到 O(s)）

> 本篇是 [02.1 · 显存、激活值与重计算](./02.1-显存、激活值与重计算.md) 的**技术专题**（同属〔铺垫〕层）。02.1 §5 讲了"重计算 = 用算力换激活显存"，并说 FlashAttention 是它"在注意力上的名场面"。这里把这句话**完整展开**：从背景、问题、数学原理讲到 Megatron 里的具体实现。
>
> 建议在 [02.1](./02.1-显存、激活值与重计算.md)（重计算）和 [02.0 §2](./02.0-Transformer与MoE结构基础.md)（注意力结构）之后读；本篇也是 [02.7 §3 Ring Attention](./02.7-上下文并行.md) 的数学前置。
>
> 面向初学者：只需要"softmax、矩阵乘、以及 02.1 的重计算直觉"。

## 0. 记号约定

| 符号 | 含义 |
|------|------|
| `N` / `d` | 序列长度（本篇 prose 里也写作 `s`）/ 每个注意力头的维度（head dim） |
| `Q, K, V` | 查询/键/值矩阵，均为 `[N, d]`（单头视角），存在 HBM |
| `S = QKᵀ/√d` | 注意力打分矩阵，`[N, N]` |
| `P = softmax(S)` | 注意力权重矩阵（逐行 softmax），`[N, N]` |
| `O = P·V` | 注意力输出，`[N, d]` |
| `M` | 片上 SRAM 容量（元素数） |
| `Br / Bc` | 行块/列块大小：论文取 `Bc = ⌈M/4d⌉`、`Br = min(⌈M/4d⌉, d)` |
| `Tr / Tc` | 行块数 `Tr = ⌈N/Br⌉`（切 Q）、列块数 `Tc = ⌈N/Bc⌉`（切 K/V） |
| `m / ℓ` | 每行的运行最大值（rowmax）/ 运行指数和（rowsum），各 `[N]` |
| **HBM / SRAM** | GPU 高带宽显存（大、慢）/ 片上共享内存（小、极快） |

> 全篇以**单头、单 batch** 视角写；多头/batch 只是把同样的计算并行复制。记号对齐 **FlashAttention 论文**（Dao et al., 2022, [arXiv:2205.14135](https://arxiv.org/abs/2205.14135)），下文 §3.2 直接给出论文 **Algorithm 1** 的伪代码。

## 1. 背景：标准注意力为什么又慢又占显存

一个注意力头的标准三步（[02.0 §2](./02.0-Transformer与MoE结构基础.md)）：

```
① S = QKᵀ/√d      # [s,d]·[d,s] → [s,s]   打分
② P = softmax(S)   # 逐行 softmax           → [s,s]   归一
③ O = P·V          # [s,s]·[s,d] → [s,d]   加权求和
```

问题出在中间那两个 **`[s, s]` 的大矩阵 `S` 和 `P`**：

- **显存**：`S`、`P` 都是 `O(s²)`。序列 `s=8K` 时，单头单条序列的 `P` 就是 `8192² ≈ 6700 万` 个元素；乘上 batch、头数、层数，激活显存直接爆（回指 [02.1 §2.2](./02.1-显存、激活值与重计算.md)：激活 ∝ `b·s·h·L`，注意力里还额外 ∝ `s²`）。
- **速度（关键但反直觉）**：现代 GPU 的算力（FLOPs）远远过剩，真正的瓶颈是**把数据在 HBM 和 SRAM 之间搬来搬去（IO）**。标准实现把 `S`、`P` 这两个 `s×s` 巨物**整个写回 HBM 再读出来**（softmax 读一遍 `S`、写一遍 `P`，`PV` 又读一遍 `P`）。论文把标准注意力的 HBM 访问量记为 **`Θ(Nd + N²)`**——`N²` 项就来自这两个 `s×s` 中间矩阵。注意力是典型的 **memory-bound（访存受限）** 算子——时间几乎全花在搬 `s²` 数据上，而非算数。

> 一句话背景：`s×s` 中间矩阵既撑爆显存，又制造 `Θ(N²)` 的 HBM 读写，是长序列注意力的两座大山。FlashAttention 是一个 **IO-aware（访存感知）** 的**精确**注意力算法（不是近似）。

---

## 2. 问题：能不能"不物化整个 s×s 矩阵"

标准实现的根子在于**先算完整的 `S`、`P`，再算 `O`**——必须把 `s×s` 落到 HBM。两个诉求：

1. **省显存**：前向别保存 `O(s²)` 的 `S`/`P`（否则反向还要用，长期占显存）。
2. **省 IO**：别把 `s×s` 反复在 HBM↔SRAM 之间搬。

FlashAttention 的答案：**分块（tiling）+ online-softmax + 反向重算**——把 `Q/K/V` 切成能塞进 SRAM 的小块，在片上一次算完一小块的 `S→softmax→PV` 并**增量累加**到输出，全程不把完整 `S`/`P` 写回 HBM。难点是：**softmax 需要一整行的全局信息（分母要对整行求和），怎么能分块算？** 这正是数学原理要解决的。

## 3. 数学原理

### 3.1 online-softmax：softmax 可以分块增量计算

先看一行 softmax（数值稳定版，减去最大值防溢出）。设一行打分为 $x_1,\dots,x_s$：

$$
m=\max_j x_j,\qquad \ell=\sum_{j} e^{x_j-m},\qquad \text{softmax}(x)_j=\frac{e^{x_j-m}}{\ell}.
$$

把这一行**切成若干块**依次处理。维护两个"运行统计量"：当前见过的最大值 $m$ 与当前的指数和 $\ell$。来了新的一块 $x^{(\text{new})}$，其块内最大值为 $\tilde m$、块内指数和为 $\tilde\ell=\sum e^{x^{(\text{new})}-\tilde m}$，则更新公式为：

$$
\boxed{\;m' = \max(m,\tilde m),\qquad
\ell' = e^{\,m-m'}\,\ell \;+\; e^{\,\tilde m-m'}\,\tilde\ell\;}
$$

**关键：$e^{m-m'}$ 这个"缩放因子（rescale）"把旧的 $\ell$ 修正到新基准 $m'$ 上。** 因为 $e^{x-m}=e^{x-m'}\cdot e^{m'-m}$，换基准只需乘一个标量。逐块套用，处理完整行后的 $(m,\ell)$ 与一次性算的完全相等——**softmax 的分母可以在只看过部分数据时就正确地增量累加**。

### 3.2 FlashAttention 前向：论文 Algorithm 1（分块 tiling）

有了 online-softmax，就能把注意力**分块（tiling）**算而不物化 `s×s`。论文的做法：**运行时把每行的输出 `O_i` 也保持"已归一化"，每来一个新块就 rescale 一次**——输出某行 $O=\frac{1}{\ell}\sum_j e^{x_j-m}V_j$，当 `m`、`ℓ` 从旧值更新到 `m^new`、`ℓ^new` 时，旧的 `O_i` 只需乘一个标量 `diag(ℓ^new)^{-1} diag(ℓ) e^{m−m^new}` 修正、再加上新块 `e^{m̃−m^new} P̃V_j` 的贡献。这正是下面 Algorithm 1 第 12 行。

**论文 Algorithm 1（前向，与 [arXiv:2205.14135](https://arxiv.org/abs/2205.14135) 一致）**——注意 **外层遍历 K/V 列块 `j`、内层遍历 Q 行块 `i`**：

```
要求: Q, K, V ∈ ℝ^{N×d} 在 HBM；片上 SRAM 大小 M

 1: 置块大小 Bc = ⌈M/4d⌉,  Br = min(⌈M/4d⌉, d)
 2: 在 HBM 初始化 O = (0)_{N×d},  ℓ = (0)_N,  m = (−∞)_N
 3: 把 Q 切成 Tr=⌈N/Br⌉ 个块 Q_1..Q_Tr（各 Br×d）；把 K,V 各切成 Tc=⌈N/Bc⌉ 个块（各 Bc×d）
 4: 把 O 切成 Tr 个块 O_i；把 ℓ、m 各切成 Tr 个块 ℓ_i、m_i
 5: for j = 1 … Tc do                                  # ← 外层：K/V 列块
 6:     把 K_j, V_j 从 HBM 载入 SRAM
 7:     for i = 1 … Tr do                              # ← 内层：Q 行块
 8:         把 Q_i, O_i, ℓ_i, m_i 从 HBM 载入 SRAM
 9:         片上算 S_ij = Q_i K_jᵀ                      ∈ ℝ^{Br×Bc}
10:         片上算 m̃_ij = rowmax(S_ij),  P̃_ij = exp(S_ij − m̃_ij),  ℓ̃_ij = rowsum(P̃_ij)
11:         片上算 m_i^new = max(m_i, m̃_ij)
                   ℓ_i^new = e^{m_i − m_i^new} ℓ_i + e^{m̃_ij − m_i^new} ℓ̃_ij
12:         写回 O_i ← diag(ℓ_i^new)^{-1} ( diag(ℓ_i) e^{m_i − m_i^new} O_i + e^{m̃_ij − m_i^new} P̃_ij V_j )   到 HBM
13:         写回 ℓ_i ← ℓ_i^new,  m_i ← m_i^new          到 HBM
14:     end for
15: end for
16: 返回 O
```

> 说明：① 第 9 行 `S_ij = Q_i K_jᵀ` 里的 `1/√d` 缩放论文折进了 softmax（可提前乘到 `Q`），为对齐论文这里省略。② 第 10–12 行就是 §3.1 的 online-softmax：`m̃/P̃/ℓ̃` 是**新块内**的局部统计量，`m^new/ℓ^new` 是 rescale 后的**运行**统计量，第 12 行把旧 `O_i` rescale 后加上新块贡献、并用 `diag(ℓ_i^new)^{-1}` 保持归一。③ 全程 `S_ij`、`P̃_ij` 只是 `Br×Bc` 小块、只在 SRAM，**完整的 `S`、`P` 从不出现在 HBM**。

**IO 复杂度（论文核心结论）**：FlashAttention 的 HBM 访问量是 **`O(N²d²·M^{-1})`**，而标准注意力是 **`Θ(Nd + N²)`**。因为通常 `d`（64–128）与 `M`（约上百 KB）满足 `d²/M ≪ 1`，FlashAttention 的 HBM 访问**远少于**标准实现——这（而非省 FLOPs）才是它更快的根因。额外显存只有 `O(N)`（存 `ℓ`、`m`/`L`）。

> **v1 vs v2 的循环顺序**：上面是 **FlashAttention-1** 的 Algorithm 1——外层 K/V、内层 Q，代价是同一个 `O_i, ℓ_i, m_i` 会被**反复读写 HBM `Tc` 次**（每个 `j` 一次）。**FlashAttention-2**（[arXiv:2307.08691](https://arxiv.org/abs/2307.08691)）把两层循环**对调**（外层 Q、内层 K/V），让 `O_i` 在内层 K/V 循环里**常驻 SRAM 累加、最后只写一次 HBM**，并只存 `logsumexp L = m + log ℓ` 一个统计量；这就是 §3.3 反向、以及 [02.7 Ring Attention](./02.7-上下文并行.md)「固定 Q、KV 流过」所采用的顺序。两者数学等价，v2 只是更省 HBM 读写、并行度更好。

### 3.3 反向：不存 P，靠重算（这就是 02.1 §5 的重计算）

反向要 `dQ/dK/dV`，公式里都要用到前向的 `P`。FlashAttention **不保存 `P`**（那是 `O(s²)`），只保存：输出 `O`（`O(s·d)`，本来就要留）和每行的 **logsumexp 统计量 `L = m + log ℓ`**（`O(s)`）。反向时用 `Q,K,V,L` **分块重算** `P_ij = exp(S_ij − L_i)`，再算梯度：

$$
dV_j = \sum_i P_{ij}^\top\, dO_i,\quad
dP_{ij}=dO_i V_j^\top,\quad
dS_{ij}=P_{ij}\odot\big(dP_{ij}-D_i\big),\quad D_i=\text{rowsum}(dO_i\odot O_i),
$$

再由 $dS$ 得 $dQ,dK$。这就是 [02.1 §5](./02.1-显存、激活值与重计算.md) 说的**"用一次额外计算换激活显存"**：注意力激活从 `O(s²)` 降到 **`O(s)`**（只留 `O` 与 `L`）。

> 小结数学：**online-softmax 让 softmax 可分块**（3.1）→ **分块 tiling、运行归一化 `O_i`**（3.2，论文 Algorithm 1）→ **反向靠 `L` 重算 P**（3.3）。三者合起来 = 全程不物化 `s×s`，激活 `O(s²)→O(s)`，HBM 访问从 `Θ(Nd+N²)` 降到 `O(N²d²/M)`（`M`=SRAM 容量，`d²/M≪1` 故远少于标准实现）。

## 4. 具体实现：Megatron 里怎么开 FlashAttention

### 4.1 训练路径：委托给 Transformer Engine（TE），不自己实现

Megatron **训练前向不自己写 FlashAttention 内核**，而是把 `core_attention` 交给 **Transformer Engine**：

- `TEDotProductAttention`（`megatron/core/extensions/transformer_engine.py:1583`）继承 `te.pytorch.DotProductAttention`；TE 内部再派发到它自己的 FlashAttention / cuDNN fused / unfused 内核。Megatron 只负责**选后端**。
- 朴素回退实现 `DotProductAttention`（`megatron/core/transformer/dot_product_attention.py`）是**纯 PyTorch 的 softmax-matmul**（会物化 `s×s`），**不是** FlashAttention，且不支持上下文并行（CP）。

### 4.2 用 `--attention-backend` 选后端

真正的开关是 **`--attention-backend`**（`TransformerConfig.attention_backend`，`transformer_config.py:143`，**默认 `auto`**），取值 `flash / fused / unfused / local / auto`。它经 `_set_attention_backend()`（`language_module.py:108`）翻译成 TE 读取的环境变量：

| `--attention-backend` | 行为 | 设置的环境变量 |
|----|----|----|
| `auto`（默认） | 让 TE 自己挑最优（flash/fused/unfused 都允许） | `NVTE_FLASH_ATTN=1, NVTE_FUSED_ATTN=1, NVTE_UNFUSED_ATTN=1` |
| `flash` | 强制 FlashAttention | `NVTE_FLASH_ATTN=1`，另两个=0 |
| `fused` | 强制 cuDNN fused attention | `NVTE_FUSED_ATTN=1`，另两个=0 |
| `unfused` | 强制未融合（会物化 `s×s`） | `NVTE_UNFUSED_ATTN=1`，另两个=0 |
| `local` | 走 mcore 朴素 PyTorch 实现 | 全 0；需配 `--spec local` |

- **`--use-flash-attn`** 是**遗留开关**（`arguments.py:2581`），实际后端选择已由 `--attention-backend` 接管；确定性模式下还禁止它（`arguments.py:1524`）。
- 注意：`local` 后端不支持 CP（长序列/超长上下文必须用 TE 的 flash/fused，见 [02.7](./02.7-上下文并行.md)）。

### 4.3 推理/解码路径：直接调 flash_attn 系列内核

推理（尤其是 flash-decode）**直接调用外部 `flash_attn` 库**，不经 TE：`attention.py` 导入 `flash_attn`（FA2）、`flash_attn_3`/`flashattn_hopper`（FA3）、`flash_attn.cute`（FA4）、`flash_mla`（MLA 专用），在 `flash_decode()`（`attention.py:721`）里用 `flash_attn_with_kvcache(...)` 完成解码期的增量注意力。

> 对上号：本篇 §3 的"分块 + online-softmax + 反向重算"就是 TE / `flash_attn` 内核内部干的事；Megatron 的角色是**选后端 + 喂配置**。想理解它和长序列并行的接力，见 [02.7 §3](./02.7-上下文并行.md)（Ring Attention 复用 online-softmax 沿序列跨卡累加）与 [02.9 §2](./02.9-进阶专题.md)（长序列工具箱）。

## 5. 版本演进：FlashAttention v1 → v4

四代的**数学核心始终不变**——tiling 分块 + online-softmax + 反向重算（§3）；每一代真正在做的，是**当算法搬到更新的 GPU 上、发现没吃满新硬件的能力时，针对新硬件特性重写 kernel**。

| 版本 | 论文 / 时间 | 目标硬件 | 一句话核心 | 峰值利用率 |
|------|------------|----------|-----------|-----------|
| **v1** | [arXiv:2205.14135](https://arxiv.org/abs/2205.14135)，2022 | Ampere(A100) 等通用 | IO 感知：tiling + online-softmax + 重算，不物化 `s×s` | 训练比标准快 2–4× |
| **v2** | [arXiv:2307.08691](https://arxiv.org/abs/2307.08691)，2023 | Ampere(A100) | 更好并行 + 更少非 matmul 开销 | A100 ~50–73%（~230 TFLOP/s） |
| **v3** | [arXiv:2407.08608](https://arxiv.org/abs/2407.08608)，2024 | Hopper(H100) | 异步（warp-spec/TMA）+ FP8 低精度 | H100 ~75–85%（740–840 TFLOP/s），FP8 ~1.2–1.3 PFLOP/s |
| **v4** | [arXiv:2603.05451](https://arxiv.org/abs/2603.05451)，代码 2025 / 论文 2026-03 | Blackwell(B200) | 破"非对称硬件扩展"：软件 exp + 条件 rescale + 全异步流水线 | 比 cuDNN SOTA 快 ~20% |

### v1（2022）— IO 感知的精确注意力
- **要解决**：标准注意力把 `s×s` 的 `S/P` 反复写读 HBM（`Θ(Nd+N²)` 访问），访存受限。
- **做法**：本篇 §3 的全部——tiling + online-softmax + 反向重算，HBM 访问降到 `O(N²d²/M)`，激活 `O(s²)→O(s)`；循环**外层 K/V、内层 Q**（Algorithm 1）。
- **效果**：BERT/GPT 训练 2–4× 加速、支持更长上下文，且是**精确**注意力（非近似）。

### v2（2023）— 榨干 Ampere 的并行度
- **要解决**：v1 GPU 利用率仅 ~25–40%；softmax 的**非矩阵乘运算**和 warp 间分工拖后腿（A100 上 Tensor Core matmul 比非 matmul 快约 16×，非 matmul 很"贵"）。
- **三个改进**：
  1. **减少非 matmul 运算**：online-softmax 中途不反复归一，**只在最后统一乘一次 `diag(ℓ)^{-1}`**，且只存 `logsumexp L`（即 §3.2 的 v2 顺序）。
  2. **增加并行维度**：除 batch、head 外，**再沿序列长度并行**，长序列 / 小 batch 下也能占满 SM。
  3. **改 warp 分工**：v1 把 K/V 切给不同 warp（split-K，需经 shared memory 通信同步）→ v2 把 **Q 切给不同 warp（split-Q，无需 warp 间通信）**，减少 shared memory 读写。
- **效果**：比 v1 快约 2×，A100 上达理论峰值的 50–73%。

### v3（2024）— 用上 Hopper 的异步与 FP8
- **要解决**：v2 在 H100 上只有 ~35% 利用率，没用到 Hopper 的异步 Tensor Core（WGMMA）、TMA、FP8。
- **三个技术（针对 Hopper）**：
  1. **warp-specialization + TMA 异步**：部分 warp 专做 GEMM（WGMMA）、部分并发做 softmax/scaling，以**生产者-消费者**方式重叠计算与数据搬运。
  2. **matmul 与 softmax 交错（ping-pong scheduling）**：让 Tensor Core 不等 softmax，藏住延迟。
  3. **FP8 低精度**：block quantization + incoherent processing 压低 FP8 精度损失（比基线数值误差低 2.6×）；FP16 的 softmax 中间量仍走 FP32，误差与 v2 相同。
- **效果**：H100 上比 v2 快 1.5–2.0×，BF16 达 740–840 TFLOP/s（75–85% 利用率），FP8 近 1.2–1.3 PFLOP/s。

### v4（代码 2025 / 论文 2026-03）— 应对 Blackwell 的"非对称扩展"
- **要解决**：Blackwell(B200) **Tensor Core 吞吐翻倍**（FP16 2.25 vs Hopper 1 PFLOPS），但 **shared memory 带宽、指数单元（SFU）、ALU 扩展慢或不变** → 瓶颈从 matmul 挪到**非 matmul 单元**。新硬件特性：TCGEN05 指令、Tensor Memory、全异步 MMA。
- **核心技术**：
  1. **软件模拟 exp()**：不走已成瓶颈的 SFU，改用 FMA 单元上的**多项式近似**算指数，把指数运算挪到 Blackwell 富余的通用计算上。
  2. **条件式 softmax rescale**：仅当新 rowmax **大到足以威胁数值稳定**时才 rescale，rescale 操作约减 10×（一个 warp 内若无线程需要 rescale，就整体跳过）。
  3. **重设计的全异步流水线**（前向 + 反向）：利用全异步 MMA + 更大 tile 最大化 Tensor Core / softmax / 访存的重叠——比 v3 的 ping-pong 复杂得多。
  4. **反向 2-CTA MMA**：反向受 shared memory 带宽限，两个 CTA 配对共享 operand B、各存半个 accumulator，约**减半 operand B 的 shared memory 流量**。
- **工程**：全部用 **CuTe-DSL**（CUTLASS 的 Python 嵌入 DSL）实现，编译快 20–30×（秒级）。
- **效果**：比 cuDNN 的 SOTA 注意力 kernel 快约 20%（BF16 比 Triton 实现 up to 1.3×）；但**非普适最快**——短序列（1–2K）/ 因果场景下 cuDNN 可能反超。

> 一条主线：**算法（§3）四代未变，变的是"怎样在具体 GPU 上把它跑到峰值"**——v1 省 IO、v2 榨 Ampere 并行、v3 用 Hopper 异步+FP8、v4 破 Blackwell 非对称瓶颈。Megatron 训练用的 TE / `flash_attn` 内核会随硬件选到对应代际的实现（§4）。

---

## 6. 小结

- **背景**：标准注意力的 `S`、`P` 是 `[s,s]` 巨物 → 激活 `O(s²)` + `O(s²)` 的 HBM 读写；注意力是访存受限（memory-bound）。
- **问题**：能否不物化完整 `s×s`？难点是 softmax 需要整行全局信息。
- **数学（对齐论文 Algorithm 1）**：**online-softmax** 用运行 `max/sum`（`m`、`ℓ`）+ rescale 因子 `e^{m−m^new}` 让 softmax 可分块增量；分块 tiling **外层 K/V、内层 Q**，第 12 行把已归一化的 `O_i` rescale 后累加新块；反向只存 `O` 与 `logsumexp L`、**分块重算 P**。IO 访问从标准的 `Θ(Nd+N²)` 降到 **`O(N²d²/M)`**，激活 `O(s²)→O(s)`。
- **实现**：Megatron 训练**委托 TE**（`TEDotProductAttention`），用 **`--attention-backend`**（默认 `auto`，可选 `flash`）→ `NVTE_*` 环境变量选后端；`--use-flash-attn` 为遗留；`local` 回退是朴素实现、不支持 CP；推理路径直接调 `flash_attn`/`flash_attn_3`/`flash_mla`。
- **一句话**：FlashAttention = 02.1 重计算思想在注意力上的极致工程化——**分块算、不落 `s×s`、反向重算**，本质是**用 IO 感知换速度**。

> 📄 **论文出处**：本篇 §3 的算法主体（Algorithm 1）出自 **FlashAttention v1**（Dao, Fu, Ermon, Rudra, Ré, NeurIPS 2022, [arXiv:2205.14135](https://arxiv.org/abs/2205.14135)）。**v2**（[arXiv:2307.08691](https://arxiv.org/abs/2307.08691), 2023）、**v3**（Shah 等，含 Tri Dao，NeurIPS 2024, [arXiv:2407.08608](https://arxiv.org/abs/2407.08608)）、**v4**（Tri Dao 等，2026, [arXiv:2603.05451](https://arxiv.org/abs/2603.05451)）四代的演进与各自贡献见 **§5**。
