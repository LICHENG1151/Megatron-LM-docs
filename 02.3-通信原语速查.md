# 02.3 · 通信原语速查（Broadcast … All-to-All）

> 本篇是 [02 · 并行化子系统](./02-并行化子系统.md) 的**铺垫基础文档**：所有并行策略（TP/PP/DP/CP/EP）的通信，归根结底都是由几个**集合通信原语**拼出来的。先把这些"积木"逐个看清——**数据从哪来、到哪去、传多少**——后面看 Megatron 的通信代码就不会晕。
>
> 建议读完 [02.1](./02.1-显存、激活值与重计算.md)（为什么要并行）后读本篇，再进入 [02.4](./02.4-并行组构建与通信详解.md)（Megatron 怎么建通信组）。集合通信 vs 点对点通信（NCCL vs RPC）的深入对照见 [02.4 §0.3 Remark](./02.4-并行组构建与通信详解.md)，本篇聚焦原语本身。

---

## 0. 怎么看下面的图示

本篇统一用 **4 卡（`G0`–`G3`）** 的「**通信前 → 通信后**」代码块图。每张卡 `Gi` 占一行，`[ ]` 里是它**当前手上的数据**；字母（`a/b/c/d`、`A/B/C/D` 或 `D0..D3`）标出这份数据**最初属于哪张卡**，`+` 表示跨卡求和的结果。归约类（Reduce / Reduce-Scatter / All-Reduce）会**额外标注"加法在哪张卡上完成"**。

看每个原语，只需盯住四件事：**① 通信前每卡有什么 → ② 通信后每卡有什么 → ③ 数据怎么走 → ④（若求和）在哪张卡上算。**

记号：`N` = 组内 GPU 数（下文取 `N=4`）；`S` = 一份数据（一个分片/一条消息）的大小；下表"每卡通信量"给的是**量级**（Ring 算法下的收发总量），用于横向比较，不是精确常数。

## 1. 一对多 / 多对一（分发与收集）

| 原语 | 语义（数据去哪） | 每卡通信量 | 典型用途 |
|------|-----------------|:--:|----------|
| **Broadcast** | root 把**同一份** D 发给组内所有卡 | `(N−1)·S` | 加载/分发权重，组内同步同一个 batch |
| **Scatter** | root 把一个大数组**切成 N 片**，第 i 片发给第 i 卡（各不相同） | `(N−1)·S` | 数据/参数分发 |
| **Gather** | Scatter 的逆：每卡的分片**汇总到 root** | `(N−1)·S` | 收集分散的输出 |
| **Reduce** | 各卡的数据**求和**（或 max…），**只有 root** 拿到结果 | `(N−1)·S` | 汇总标量/统计量 |

下面统一以 **4 卡（root = G0）** 画图。

**Broadcast** —— root 把同一份 `D` 复制给所有卡（无计算）：

```
通信前              通信后
G0(root): [D]       G0: [D]
G1:       [ ]       G1: [D]
G2:       [ ]       G2: [D]
G3:       [ ]       G3: [D]

路径：G0 沿树/环把 D 逐跳分发（约 log N 步），只搬运、不求和。
      G0 ─D─► G1 ─D─► G2 ─D─► G3   （逻辑示意）
```

**Scatter** —— root 把大数组切 4 片，第 i 片发给 Gi（各不相同，无计算）：

```
通信前                       通信后
G0(root): [D0 D1 D2 D3]      G0: [D0]
G1:       [ ]                G1: [D1]
G2:       [ ]                G2: [D2]
G3:       [ ]                G3: [D3]

路径：G0 把 Di 分别点发给 Gi，只搬运、不求和。
```

**Gather**（Scatter 的逆）—— 各卡的片汇到 root，**拼接不求和**：

```
通信前           通信后
G0: [D0]         G0(root): [D0 D1 D2 D3]
G1: [D1]         G1:       [D1]
G2: [D2]         G2:       [D2]
G3: [D3]         G3:       [D3]

路径：G1/G2/G3 把 Di 发回 G0，只拼接、不求和。
```

**Reduce** —— 所有卡求和，**只有 root 拿到和**：

```
通信前           通信后
G0(root): [a]    G0: [a+b+c+d]   ← 只有 root 得到总和
G1:       [b]    G1: [b]（不变）
G2:       [c]    G2: [c]（不变）
G3:       [d]    G3: [d]（不变）

求和在哪算（tree/ring，边传边加 reduce-on-receive，非"先全搬到 root 再算"）：
  G3 ─d─────────► G2:  算 c+d          ← 加法在 G2
  G2 ─(c+d)─────► G1:  算 b+(c+d)      ← 加法在 G1
  G1 ─(b+c+d)───► G0:  算 a+(b+c+d)    ← 最后一次加法在 root(G0)
  即：加法沿归约树逐跳累加，最终一加落在 root 上。
```

## 2. 多对多（训练主力，务必吃透）

| 原语 | 语义（数据去哪） | 每卡通信量 | 典型用途 |
|------|-----------------|:--:|----------|
| **All-Gather** | 各持一片 → **所有卡都拿到完整拼接** | `(N−1)·S` | **SP 补齐序列**、**ZeRO-3 聚合参数** |
| **Reduce-Scatter** | 先求和、再切片：每卡只保留**和的 1/N** | `(N−1)·S` | **SP**、**分布式优化器切梯度** |
| **All-Reduce** | 求和后**所有卡拿到相同的完整结果** | **≈ 2S**（与 N 几乎无关） | **DP 同步梯度**、**TP 合并激活/部分和** |
| **All-to-All** | 每卡把**不同的块**发给每一张卡（像矩阵转置） | ∝ 数据量 | **MoE 的 dispatch / combine** |

**四卡「通信前 → 通信后」图示**（Reduce-Scatter 单独详解见 [§2.2](#22-reduce-scatter-详解求和到底在哪儿算)）：

**All-Gather** —— 各持一片 → 人手一份完整拼接（**无计算**，只搬运）：

```
通信前         通信后（人手一份完整）
G0: [A]        G0: [A B C D]
G1: [B]        G1: [A B C D]
G2: [C]        G2: [A B C D]
G3: [D]        G3: [A B C D]

路径：Ring，N−1 步，每步把手里的块传给下一邻居，绕一圈后人人集齐：
      G0 ─► G1 ─► G2 ─► G3 ─►(回) G0   （只搬运、不求和）
```

**All-Reduce** —— 求和后**所有卡**都拿到相同的完整结果：

```
通信前         通信后（人手一份完整和）
G0: [a]        G0: [a+b+c+d]
G1: [b]        G1: [a+b+c+d]
G2: [c]        G2: [a+b+c+d]
G3: [d]        G3: [a+b+c+d]

求和在哪算：= Reduce-Scatter + All-Gather，**没有中心节点**——
  ① RS 阶段：各卡负责"自己那 1/N 片"，沿环边传边加算出该片的和（加法摊到每张卡）；
  ② AG 阶段：把 4 片和拼回，人手一份（只搬运）。
（标量 a 只是示意；实际是长向量按元素求和，才能切成 N 片摊算，见 §2.1/§2.2）
```

**All-to-All** —— 人人对人人，等价矩阵**转置**（**无计算**，只重排）。`X[i→j]` = Gi 要发给 Gj 的块：

```
通信前（每卡持有发往各卡的 4 块）      通信后（每卡收齐"发给它"的 4 块 = 转置）
G0: [X0→0 X0→1 X0→2 X0→3]            G0: [X0→0 X1→0 X2→0 X3→0]
G1: [X1→0 X1→1 X1→2 X1→3]     ──►    G1: [X0→1 X1→1 X2→1 X3→1]
G2: [X2→0 X2→1 X2→2 X2→3]            G2: [X0→2 X1→2 X2→2 X3→2]
G3: [X3→0 X3→1 X3→2 X3→3]            G3: [X0→3 X1→3 X2→3 X3→3]

路径：每卡把第 j 块发给 Gj，同时收下各卡发来的块；对角线(X i→i)留本地。
```

### 2.1 一条贯穿全篇的恒等式 ★

$$
\boxed{\textbf{All-Reduce} \;=\; \textbf{Reduce-Scatter} \;+\; \textbf{All-Gather}}
$$

```
All-Reduce = ① Reduce-Scatter（各卡先各自求和自己负责的 1/N 段）
           + ② All-Gather   （再把各段拼回完整，人手一份）
```

- 这解释了为什么 **All-Reduce 的通信量 ≈ 2S、且几乎与卡数 N 无关**（Ring 实现下 RS 和 AG 各约 S）。
- 后面很多"通信量守恒"的结论都靠它：
  - **ZeRO-1/2** 把 DP 的一次 All-Reduce 拆成 RS + AG，**总通信量不变**，却把优化器状态/梯度切成 1/N 省了显存（[02.4 §2.6](./02.4-并行组构建与通信详解.md)）。
  - **序列并行(SP)** 把 TP 的 All-Reduce 换成 AG + RS，**总量不变**，却额外省下 LayerNorm/Dropout 的激活（[02.5 §8](./02.5-张量并行实现详解.md)）。

### 2.2 Reduce-Scatter 详解：求和到底在哪儿算？

Reduce-Scatter 最容易犯迷糊，这里彻底讲透。

**① 语义（精确版）**：输入是——组内**每张卡都持有一个完整长度的数组**（看成 `N` 个分片拼起来）；操作先对**所有卡的对应元素求和**（Reduce），再把结果**按分片切开、每张卡只留一片**（Scatter）。第 `r` 张卡最终只拿到"总和"的第 `r` 片，长度 = 总长 / `N`。

用 **4 卡**、每卡数组长 4（分成 a / b / c / d 四片）的小例子：

```
通信前（每卡持有完整 4 片，下标 = 卡号）：
  G0: [a0, b0, c0, d0]
  G1: [a1, b1, c1, d1]
  G2: [a2, b2, c2, d2]
  G3: [a3, b3, c3, d3]

通信后（四片分别跨卡求和，每片的和落到不同卡）：
  G0 ←  a0+a1+a2+a3      # 只拿到 "a 片" 的和
  G1 ←  b0+b1+b2+b3      # 只拿到 "b 片" 的和
  G2 ←  c0+c1+c2+c3      # 只拿到 "c 片" 的和
  G3 ←  d0+d1+d2+d3      # 只拿到 "d 片" 的和
```

> 对照 **§1 的 Reduce**：Reduce 是"所有片全求和、结果**只给 root 一张卡**"；Reduce-Scatter 是"所有片全求和，但**每片的和分给不同的卡**"——所以 RS ≈ Reduce 之后立刻把结果 Scatter 出去，名字即由此而来。

**② PyTorch 代码**（`reduce_scatter_tensor`，最直观）：

```python
import torch, torch.distributed as dist
# world_size N=4；每卡一个长度 4 的输入张量（4 片各长 1）
#   rank0 in=[a0,b0,c0,d0]  rank1 in=[a1,b1,c1,d1]  rank2 in=[...]  rank3 in=[a3,b3,c3,d3]
inp = torch.tensor(local_values, device="cuda")   # shape [N]  = [4]
out = torch.empty(1, device="cuda")               # 每卡只收 1 片(N 分之一)
dist.reduce_scatter_tensor(out, inp, op=dist.ReduceOp.SUM)
# 结束后：rank0.out=a0+a1+a2+a3；rank1.out=b0+b1+b2+b3；rank2.out=c…；rank3.out=d0+d1+d2+d3
```

**③ 核心问题：求和在哪儿算？——分散在所有卡上，"边传边加"（reduce-on-receive），没有中心节点。**

关键：Reduce-Scatter **绝不是**先把所有数据搬到某一张卡上求和（那样带宽和显存都爆）。NCCL 用 **Ring 算法**把求和**摊到全组每张卡**——数据沿环逐跳流动，**每一跳，接收方就把收到的分片加到自己本地对应的分片上**。以 `N=4`、追踪"a 片"（最终归 card0）的部分和怎么在环上滚起来：

```
目标：G0 最终拿到 a 片的完整和 a0+a1+a2+a3
环：G0 → G1 → G2 → G3 →(回) G0，每步只传一个分片

  起点   G1 手里的 a1
  第1跳  G1──a1──►G2 ：  G2 算  a1+a2         ← 加法在 G2 上
  第2跳  G2─(a1+a2)─►G3： G3 算 (a1+a2)+a3    ← 加法在 G3 上
  第3跳  G3─(a1+a2+a3)─►G0：G0 算 …+a0 = 全和  ← 最后一次加法在 G0 上
```

- 这只是"a 片"一条流水线。实际上 **b/c/d 四片的部分和同时在环上并行滚动**（错开一个身位），所以**每一步、每张卡都在收一个分片、加一次、再发一个分片**——求和的计算量**均匀摊到全组每张卡**，没有谁是瓶颈。
- 所以答案是：**求和发生在每一张接收卡上、在分片到达它的那一跳完成**；你负责的那一片的**最后一次加法**在你自己卡上做，之前的部分和是环上**上游各卡逐跳累加**出来的。
- 也正因为"边传边加、每卡只经手 (N−1)/N 份数据"，RS 的每卡通信量才只有 **≈ S**（而不是把全量搬给一张卡的 `(N−1)·S`），且计算无中心瓶颈——这是它能扩展到成百上千卡的根本。

**④ 训练里的落点**：分布式优化器 / ZeRO 反向末尾对梯度做 Reduce-Scatter 后，**每卡恰好只持有"自己负责的那 1/N 段梯度"的完整求和结果**，正好各自去更新自己那 1/N 的参数、优化器状态（[02.4 §2.6](./02.4-并行组构建与通信详解.md)、[02.3.1 FSDP](./02.3.1-FSDP.md)）；序列并行 SP 用它切激活同理（[02.5 §8](./02.5-张量并行实现详解.md)）。

### 2.3 All-to-All 直觉

每张卡都持有"发给全组每一张卡"的不同数据块，一次操作后完成"人人对人人"的重排——等价于把数据矩阵**转置**。MoE 里 token 要去往"持有其目标专家"的卡，正是这种"人人重排"，所以 EP 用两次 All-to-All（送过去 + 收回来，见 [02.8 §3](./02.8-专家并行.md)）。

> **Remark：「转置」是逻辑类比，不是"先收集到一张卡再分发"——All-to-All 无 root、峰值显存 ≈ 2S/卡**
>
> 容易误解成"先把所有块 Gather 到某张卡拼成大矩阵、再 Scatter 回去"，那样这张卡会持有 `N×S` 全量、峰值显存爆炸。**实际恰恰相反**：All-to-All 是**全直连的点对点**，没有中心节点。
>
> - **实现**：每张卡 `i` 把自己的第 `j` 块**直接发给 `Gj`**（`N−1` 个 send），同时从每张卡 `k` **直接收第 `i` 块**（`N−1` 个 recv）；发给自己的对角块 `X[i→i]` 是**本地 memcpy**、不走网络。这些收发在一个 `ncclGroupStart/End`（PyTorch `dist.all_to_all_single` 内部）里**并发提交**，NCCL 常按 `N−1` 轮 pairwise 交换（第 k 轮和伙伴 `(r±k)%N` 各换一块）把负载铺满所有链路。
> - **显存**：任何一张卡全程只需 `send_buf`（≈ `S`）+ `recv_buf`（≈ `S`）两块，**峰值 ≈ 2S、与 `N` 无关放大**；**没有任何一张卡持有过全矩阵**。"先 Gather 再 Scatter"才是那个 `O(N·S)` 的中心瓶颈，All-to-All 正是为绕开它而生。
> - **全局视角**：块 `X[i→j]` 从源卡 `i` **直飞**目的卡 `j`，从不经过中转卡；只有把全局 `N×N` 块网格摆出来看，才呈现"转置"的形状。
> - **MoE 用的是变长版 all-to-all-v**：各卡发往每个专家的 token 数不同（由 routing_map 定），`in/out_splits` 不等长；实现仍是上面的分组点对点、显存仍是两块缓冲。负载不均只让**通信量倾斜**，不会让某卡"拼全矩阵"（见 [02.8 §3](./02.8-专家并行.md)）。

### 2.4 各原语的 torch.distributed 接口（代码对照）

每个原语在 PyTorch 里都有对应函数，看代码时可对号入座（`op=` 只对归约类有效，`SUM`/`MAX`/…）：

| 原语 | PyTorch API | 关键参数 |
|------|-------------|----------|
| Broadcast | `dist.broadcast(tensor, src)` | `src`=root |
| Scatter | `dist.scatter(tensor, scatter_list, src)` | root 提供 `scatter_list` |
| Gather | `dist.gather(tensor, gather_list, dst)` | `dst`=root |
| Reduce | `dist.reduce(tensor, dst, op)` | 只有 `dst` 拿到结果 |
| All-Gather | `dist.all_gather_into_tensor(out, inp)` | `out` 是 `inp` 的 N 倍长 |
| **Reduce-Scatter** | `dist.reduce_scatter_tensor(out, inp, op)` | `out` 是 `inp` 的 1/N 长（见 §2.2） |
| All-Reduce | `dist.all_reduce(tensor, op)` | 原地，人手一份完整结果 |
| All-to-All | `dist.all_to_all_single(out, inp)` | 每卡切 N 块、各发一块 |
| Send / Recv | `dist.isend(tensor, dst)` / `dist.irecv(tensor, src)` | 点对点，配对使用 |

> Megatron 里的 TP/DP 通信最终就落到这些函数（外面再包一层 `torch.autograd.Function` 让反向也正确，见 [02.5 §3](./02.5-张量并行实现详解.md)）。

## 3. 点对点（P2P）

| 原语 | 语义 | 每卡通信量 | 典型用途 |
|------|------|:--:|----------|
| **Send / Recv** | 两张卡之间直接收发（一发一收配对） | `S` | **PP 相邻 stage 传激活/梯度** |
| **Ring Send/Recv** | N 卡排成环，每步把手里的块传给下一个邻居，`N−1` 步后每卡都见过所有块 | 每步 `S` | **Ring All-Reduce、CP 环形传 KV** 的底座 |

**Send / Recv** —— 两卡之间一发一收，其余卡不参与（**无归约**）：

```
通信前         通信后
G0: [X]        G0: [X]
G1: [ ]        G1: [X]    ← G0.send(X) ─► G1.recv()
```

**Ring Send/Recv** —— N 卡排成环，每步把手里的块传给下一邻居：

```
Ring:  G0 ─► G1 ─► G2 ─► G3 ─┐
        ▲────────────────────┘     （环形，逐跳传递，N−1 步走遍全组）
```

> Ring 是很多"看似高级"的集合通信的底层实现：All-Reduce 的 Ring 算法、CP 的 [Ring Attention](./02.7-上下文并行.md) 都是"数据沿环逐跳流动 + 边传边算"，把通信藏进计算里。

## 4. 原语 → 并行策略 速查

| 并行策略 | 主要用到的原语 | 一句话 |
|----------|---------------|--------|
| **DP**（数据并行） | All-Reduce（或 RS+AG） | 复制模型、切数据，反向末尾同步梯度 |
| **ZeRO / 分布式优化器** | Reduce-Scatter + All-Gather | 把 All-Reduce 拆开，顺带把状态切成 1/N |
| **TP**（张量并行） | All-Reduce | 合并层内被切矩阵产生的部分和 |
| **SP**（序列并行） | All-Gather + Reduce-Scatter | 替换 TP 的 All-Reduce，额外省激活 |
| **PP**（流水线并行） | Send/Recv（P2P） | 相邻 stage 传激活，通信量最小 |
| **CP**（上下文并行） | Ring Send/Recv 或 All-to-All | 注意力跨卡交换 K/V |
| **EP**（专家并行） | All-to-All ×2 | token 去目标专家卡、算完再送回 |

> 每个原语都要指定一个 **process group**（在哪个"群"里做）——群怎么按 rank 划分、怎么建，是 [02.4](./02.4-并行组构建与通信详解.md) 的主题。

---

## 5. 小结

- 所有并行通信都是这几块积木拼出来的：**分发**（Broadcast/Scatter）、**收集**（Gather/All-Gather）、**归约**（Reduce/Reduce-Scatter/All-Reduce）、**重排**（All-to-All）、**点对点**（Send/Recv、Ring）。
- 记住三件事就够用：① **All-Reduce = Reduce-Scatter + All-Gather**（通信量 ≈2S，与卡数无关）；② **多对多原语**（AG/RS/AR/A2A）是训练主力；③ 每个原语关键看"**做完后每张卡手上有什么**"。
