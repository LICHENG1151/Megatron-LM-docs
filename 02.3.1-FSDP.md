# 02.3.1 · FSDP（把 16P 模型态切成 16P/N）

> 本篇是 [02.3 · 通信原语速查](./02.3-通信原语速查.md) 的**技术专题**（同属〔铺垫〕层）：把 [02.1 §2](./02.1-显存、激活值与重计算.md) 那笔"模型态 16P"用 [02.3](./02.3-通信原语速查.md) 的 **All-Gather / Reduce-Scatter** 原语沿 DP 组切成 `16P/N`，这就是 **FSDP（Fully Sharded Data Parallel）= ZeRO-3**。
>
> 建议在 [02.1](./02.1-显存、激活值与重计算.md)（16P 显存账）和 [02.3](./02.3-通信原语速查.md)（AG/RS 原语、`AR=RS+AG`）之后读。本篇讲**通用算法 + Megatron 落地**；它与 [02.4 §2.6](./02.4-并行组构建与通信详解.md)（Megatron 的 ZeRO-1/2 分布式优化器）、[02.8 §3](./02.8-进阶专题.md)（FSDP vs TP+PP 选型）互为补充。
>
> 面向初学者：只需要 02.1 的"16P = 2P+2P+12P"和 02.3 的"AR=RS+AG"两条结论。

---

## 0. 记号约定

| 符号 | 含义 |
|------|------|
| `P` | 参数总量（个数） |
| `N` | 一个 DP（数据并行）组里的 GPU 数（分片份数） |
| `16P` | Adam 混精下的模型态字节数（[02.1 §2.1](./02.1-显存、激活值与重计算.md)：参数 2P + 梯度 2P + 优化器状态 12P） |
| **AG / RS / AR** | All-Gather / Reduce-Scatter / All-Reduce（[02.3](./02.3-通信原语速查.md)） |

---

## 1. 背景：数据并行的致命冗余

普通**数据并行（DP）**里，每张卡都存一份**完整**模型态（[02.1 §2.1](./02.1-显存、激活值与重计算.md) 的 16P），只是各喂不同数据、反向末尾 All-Reduce 同步梯度。问题是：一个 `N` 卡 DP 组，把**同样的 16P 存了 `N` 份**——纯冗余。7B 模型的 16P≈112GB，单卡已经装不下，复制 `N` 份更是浪费。

**ZeRO（Zero Redundancy Optimizer）** 的思路：模型态在 DP 组里本就人手一份、完全相同，**为什么不各存 1/N、要用时再临时聚齐？** 按"切得多狠"分三级递进：

| 级别 | 切什么 | 每卡模型态 |
|------|--------|-----------|
| **ZeRO-1** | 只切**优化器状态**（12P） | `4P + 12P/N` |
| **ZeRO-2** | +切**梯度**（2P） | `2P + 14P/N` |
| **ZeRO-3 / FSDP** | +切**参数**本身（2P） | `16P/N` |

**FSDP = ZeRO-3**：把**参数、梯度、优化器状态三者**都沿 DP 组切成 1/N，每卡常驻只有 `16P/N`——卡越多越省，近似线性。

---

## 2. 问题：参数被切了，前向怎么算

优化器状态、梯度切了不影响前向（它们只在反向/更新用）。但**参数被切了**就有矛盾：**前向/反向算某一层时，需要那一层的完整权重**，可每卡只存了 `1/N` 片。

诉求：既要"常驻只存 `16P/N`"省显存，又要"算到某层时能拿到完整权重"。FSDP 的答案：**用到哪层，就临时把那层的参数 All-Gather 聚成完整，算完立刻扔掉**——以通信换显存。难点在于**这多出来的 All-Gather 通信会不会拖垮训练**，这要靠数学账 + 重叠来回答。

---

## 3. 数学原理

### 3.1 显存账：为什么 ZeRO-3 是 `16P/N`

设 DP=`N`。把 16P 拆成"可切的"和"要复制的"两部分，逐级切分：

$$
\underbrace{2P}_{\text{参数}}+\underbrace{2P}_{\text{梯度}}+\underbrace{12P}_{\text{优化器状态}}
\;\xrightarrow[\text{ZeRO-1}]{}\; 2P+2P+\tfrac{12P}{N}
\;\xrightarrow[\text{ZeRO-2}]{}\; 2P+\tfrac{14P}{N}
\;\xrightarrow[\text{ZeRO-3}]{}\; \boxed{\tfrac{16P}{N}}
$$

`N` 越大，ZeRO-3 每卡显存越接近 0（模型态维度）——这就是 FSDP 能在很多卡上塞下超大模型的根本。

### 3.2 通信账：ZeRO-3 比 DP 多花多少

用 [02.3 的恒等式 `AR = RS + AG`](./02.3-通信原语速查.md) 来算（记一份完整参数/梯度的规模为 `Φ`）：

- **普通 DP**：反向末尾一次梯度 All-Reduce = `2Φ`（= RS+AG）。
- **ZeRO-1/2**：把这次 All-Reduce **拆成** 梯度 Reduce-Scatter（反向）+ 参数 All-Gather（更新后）= 仍是 `RS+AG = 2Φ`。**通信总量不变**，却把状态/梯度切成 `1/N`（[02.4 §2.6](./02.4-并行组构建与通信详解.md) 详解）。
- **ZeRO-3 / FSDP**：因为参数也切了、且**算完即弃**，所以：
  - **前向**：逐层 All-Gather 完整参数（用完丢）= `+1Φ`（DP 没有这项）。
  - **反向**：参数已丢，**再 All-Gather 一次** = `+1Φ`；梯度 Reduce-Scatter = `1Φ`。

合计 ZeRO-3 ≈ `3Φ`，**约为 DP（`2Φ`）的 1.5 倍**——多出来的正是"反向前那次重新聚参"。这就是 02.8 §3 说"FSDP 通信更重"的来源。

> 关键权衡：ZeRO-3 用 **≈1.5× 通信**换来**模型态 `16P/N`** 的显存。所以 FSDP 好不好用，取决于这多出的通信**能否被计算掩盖**（[02.8 §3.2](./02.8-进阶专题.md) 的"通信隐藏度 80%"判据）。

### 3.3 flat buffer + 预取重叠：把聚参藏进计算

两个让 FSDP 通信高效的工程手段：

- **flat buffer（扁平缓冲）**：把一层（或一组）参数 **flatten 成一维**再按 DP 切片，All-Gather/Reduce-Scatter 一次搬一大片连续内存，而非零散小张量——通信效率高。
- **预取重叠（prefetch overlap）**：**算当前层时，异步 All-Gather 下一层的完整参数**；等算到下一层时它已就位。梯度则边算边 Reduce-Scatter。理想情况下聚参/散梯度的通信全被计算盖住（"隐藏度→100%"），FSDP 就几乎"免费"。

## 4. 具体实现：Megatron 里怎么开 FSDP

### 4.1 两套 FSDP 实现

Megatron 在 `megatron/core/distributed/` 下提供**两条** FSDP 路线：

| 实现 | 开关 | 说明 | 关键类/文件 |
|------|------|------|------|
| **Torch FSDP2** | `--use-torch-fsdp2` | 薄封装 PyTorch 原生 `fully_shard`，需 **torch ≥ 2.4** | `TorchFullyShardedDataParallel`（`torch_fully_sharded_data_parallel.py:28`） |
| **Megatron-FSDP** | `--use-megatron-fsdp` | NVIDIA 自研的 native-PyTorch FSDP 库（可 pip 装 `megatron-fsdp`） | `FullyShardedDataParallel`(adapter, `fsdp/mcore_fsdp_adapter.py:62`) → `MegatronFSDP`(`fsdp/src/megatron_fsdp/megatron_fsdp.py:106`) |

选择逻辑（`models/dist_utils.py:214`）：`use_megatron_fsdp` → Megatron-FSDP；否则 `use_torch_fsdp2` → Torch FSDP2；都没开 → 普通 `DistributedDataParallel`。两个 FSDP 开关**互斥**。

### 4.2 用 `--data-parallel-sharding-strategy` 选 ZeRO 档位

一个开关贯穿 §1 的 ZeRO-1/2/3（`MegatronFSDP` docstring `megatron_fsdp.py:113`）：

| `--data-parallel-sharding-strategy` | 切什么 | 相当于 |
|----|----|----|
| `no_shard` | 都不切（普通 DP） | 基线 |
| `optim` | 只切优化器状态 | ZeRO-1 |
| `optim_grads` | +切梯度 | ZeRO-2 |
| `optim_grads_params`（**CLI 默认**） | +切参数 | **ZeRO-3** |

（Torch FSDP2 侧则用 `--torch-fsdp2-no-reshard-after-forward` 从 ZeRO-3 退到 ZeRO-2 档；HSDP 分层切分用 `--outer-dp-sharding-strategy`。）

### 4.3 与 DistributedOptimizer（ZeRO-1）的关系

Megatron 还有一条**不走 FSDP 的 ZeRO-1**：**分布式优化器** `DistributedOptimizer`（`--use-distributed-optimizer`，`distrib_optimizer.py:107`），它挂在普通 DDP 路径上，**只切优化器状态**、参数与梯度仍复制——是 Megatron 默认推荐的轻量起点（[02.4 §2.6](./02.4-并行组构建与通信详解.md) 详解）。三者是同一"切分谱系"上的不同档位：

```
DistributedOptimizer(ZeRO-1，只切优化器状态) ── FSDP optim_grads(ZeRO-2) ── FSDP optim_grads_params(ZeRO-3，全切)
        通信 ≈ DP                                                              通信 ≈ 1.5× DP
```

注意：`--use-distributed-optimizer` 与 `--use-torch-fsdp2` **互斥**（`arguments.py:1004`）。

### 4.4 使用约束（重要）

- **Torch FSDP2**：不支持流水线并行（PP）、专家并行（EP）、MCore 分布式优化器、梯度累加融合；需 `--ckpt-format torch_dist`（或 `torch_dcp`）+ `--untie-embeddings-and-output-weights`；不支持 fp16。
- **Megatron-FSDP**：需 `--ckpt-format fsdp_dtensor`；官方推荐组合 `--use-megatron-fsdp --ckpt-format fsdp_dtensor --init-model-with-meta-device`。
- 用例见 `examples/megatron_fsdp/`。

> 回指：**要不要用 FSDP** 是选型问题（FSDP 提 MFU vs TP+PP 关重通信进机内 NVLink），判据见 [02.8 §3](./02.8-进阶专题.md)；FSDP 的 ZeRO-1/2 底层机制（DDP 分桶、Reduce-Scatter+All-Gather 切分）见 [02.4 §2.6](./02.4-并行组构建与通信详解.md) 与 [04 §2.2](./04-分布式训练与优化器.md)。

## 5. 小结

- **背景**：普通 DP 把 16P 模型态在 `N` 卡上冗余存 `N` 份 → 浪费。ZeRO 各存 1/N、用时聚齐。
- **问题**：ZeRO-3 连参数也切了，前向算某层却要完整权重 → 用到才 All-Gather、算完即弃。
- **数学**：显存 `16P → 16P/N`（ZeRO-1 `4P+12P/N` → ZeRO-2 `2P+14P/N` → ZeRO-3 `16P/N`）；通信用 `AR=RS+AG` 一算，ZeRO-3 ≈ `3Φ` = **1.5× DP**（多一次反向前聚参）；靠 **flat buffer + 预取重叠** 把聚参藏进计算。
- **实现**：Megatron 提供 **Torch FSDP2**（`--use-torch-fsdp2`）和 **Megatron-FSDP**（`--use-megatron-fsdp`）两套；`--data-parallel-sharding-strategy`（默认 `optim_grads_params`=ZeRO-3）选档位；**DistributedOptimizer=ZeRO-1** 是 DDP 路径上的轻量起点；注意 ckpt 格式与 PP/EP 约束。
- **一句话**：FSDP = 用 [02.3](./02.3-通信原语速查.md) 的 AG/RS 把 [02.1](./02.1-显存、激活值与重计算.md) 的 16P 全切成 `16P/N`，以 ≈1.5× 通信换近线性显存节省。
