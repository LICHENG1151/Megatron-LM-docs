# 02.7 · 上下文并行（Context Parallel, CP）

> 本篇是 [02 · 并行化子系统](./02-并行化子系统.md) 的**子文档**，与 [02.4](./02.4-并行组构建与通信详解.md)（通信组）、[02.5](./02.5-张量并行实现详解.md)（TP）、[02.6](./02.6-流水线并行与1F1B调度.md)（PP）并列，讲**上下文并行 CP**——沿序列维切超长序列。姊妹篇 [02.8 · 专家并行 EP](./02.8-专家并行.md)（切 MoE 专家）单独成篇。CP 组同样来自 02.4 的 `parallel_state`。
>
> 相关源码：`megatron/core/transformer/{dot_product_attention,transformer_config}.py`、`parallel_state.get_context_parallel_group()`。

## 1. CP 在切什么，难点在哪

TP 切层内矩阵、PP 切层深度，**CP 切的是「序列长度 s」**——把一条超长序列（如 128K token）沿序列维分成若干段，每个 CP rank 只持有一段。

```
序列 [t0 ... t127k]  CP=4:
  cp_rank 0: t[0:32k]
  cp_rank 1: t[32k:64k]
  cp_rank 2: t[64k:96k]
  cp_rank 3: t[96k:128k]
```

**难点在注意力**：MLP、LayerNorm 都是逐 token 的，切开序列后各算各的、互不影响；但 **Attention 要求每个 token 都能看到（attend to）所有位置的 K/V**。序列被切开后，本地 rank 只有部分 K/V，必须**跨 CP 组交换 K/V** 才能算完整的注意力。这就是 CP 的全部通信来源。

> ⚠️ 别和 **SP（序列并行）** 搞混：SP 也「沿序列切」，但它是 TP 的省显存附庸、进注意力前会把序列 all-gather 拼回完整；CP 才是独立并行维、注意力全程切着算。两者的实现与多机对比见 **§5**。

## 2. 四种 CP 通信方式（`cp_comm_type`）

由 `transformer_config.py:891` 的 `cp_comm_type` 选择，注意力计算本身委托给 Transformer Engine 完成（`dot_product_attention.py:58` 处仅当 `context_parallel_size==1` 才走纯本地路径）：

| `cp_comm_type` | 机制 | 通信原语 | 特点 |
|----------------|------|----------|------|
| **`p2p`** | Ring Attention：KV 分块在环形拓扑中逐跳传递，每收到一块就算一次部分注意力，用 online-softmax 累加 | P2P（`send/recv`，异步） | 通信可与计算重叠，最常用 |
| **`all_gather`** | 注意力前先 All-Gather 出完整 KV 序列 | All-Gather | 实现简单，显存/带宽开销较大 |
| **`a2a`** | DeepSpeed-Ulysses 式：用 All-to-All 把**注意力头**散到 CP 组，让每个 rank 拿到「全序列、部分头」 | All-to-All | 通信量小，受头数约束 |
| **`a2a+p2p`** | 分层组合：组内 a2a + 组间 p2p ring | A2A + P2P | 超大 CP 的层次化方案 |

> CP 组由 `parallel_state.get_context_parallel_group()` 提供（02.4 §1.4 的 `_CONTEXT_PARALLEL_GROUP`）。`all_gather` 之外，CP 通常与 DP 合并成 `dp-cp` 组参与梯度同步（02.4 中 `get_ranks('dp-cp')`）。

## 3. Ring Attention（`p2p`）的计算-通信顺序

以 CP=4 为例，每个 rank 持有自己那段 Q，并轮流接收其他 rank 的 KV：

```
本地: Q_i, K_i, V_i （第 i 段）
out_i = 0; 在线 softmax 统计量 (m, l) 初始化
for step in range(CP):                       # 环形 CP 跳
    用当前持有的 (K_j, V_j) 算一次"部分注意力"
    online_softmax 累加到 out_i              # 不需要完整 KV 也能正确累加 ★
    异步 P2P：把 K_j,V_j 发给环上下一个 rank，同时从上一个 rank 收下一块   [通信]
out_i 即为 Q_i 对全序列的完整注意力
```

关键洞见：**online-softmax（FlashAttention 的核心技巧）让注意力可以分块累加**，所以不必一次性持有全部 KV——这正是 Ring Attention 能成立的数学基础。P2P 异步发起，KV 传输与上一块的注意力计算重叠，把通信藏进计算里。

> online-softmax 的严谨递推（运行 `max/sum` + rescale 因子）见 [02.1.1 · FlashAttention §3](./02.1.1-FlashAttention.md)；Ring Attention 就是把它从"单卡分块"推广到"沿序列跨卡累加"。

## 4. 因果掩码下的负载均衡（Zigzag / Striped）

Ring Attention 有个容易被忽略的坑：**因果掩码（causal mask）会让各 CP rank 的计算量严重不均**。

**问题**：自回归语言模型里，第 `t` 个 token 只能 attend 到 `≤ t` 的位置。若把序列**按顺序**平均切给各 rank（rank 0 拿最前一段、rank N−1 拿最后一段），那么：

```
rank 0（最前段）：token 都很靠前，能看的历史少 → 计算量最小、很快算完就空等
rank N-1(最后段)：token 都很靠后，要 attend 几乎整条序列 → 计算量最大、拖慢全环
```

于是"顺序切"让**靠后的 rank 过载、靠前的 rank 闲置**，整个 Ring 被最慢的那张卡拖住。

**招法 —— Zigzag（之字形）切分**：把序列切成 `2N` 个小块（N=CP 卡数），让 **rank i 同时持有第 `i` 块和第 `2N−1−i` 块**（一前一后配对）。这样每张卡都"一段靠前 + 一段靠后"，因果计算量大致拉平：

```
CP=4, 序列切成 8 块 [0 1 2 3 4 5 6 7]：
  rank0: 块0 + 块7      rank1: 块1 + 块6
  rank2: 块2 + 块5      rank3: 块3 + 块4      ← 每卡都是"一头一尾"，负载均衡
```

**Striped（条纹）** 是另一种交错切法（按 stride 把 token 打散分配），目标相同——都是让每张卡的因果工作量尽量相等。

![因果注意力的负载均衡：Zigzag 与 Striped 两种切分](./images/causal_workload.png)

> 直觉：因果掩码让"工作量"沿序列递增，所以任何"把递增曲线均分给各卡"的交错方式都能均衡负载。Megatron 的 CP 在处理因果注意力时正是用这类重排（配合 [02.9 §2.2](./02.9-进阶专题.md) 的 2D CP 一起支撑超长序列）。

## 5. CP vs SP：两种「沿序列切」的区别（附多机多卡对比）

初学者最容易混的一点：**SP（序列并行）也沿序列切**，那它和 CP 是不是一回事？**不是**。SP 是 TP 的省显存附庸、寄生在 TP 组上；CP 是独立的并行维、有自己的组。分水岭在**怎么对待注意力**。

### 5.1 SP 的具体实现（复述要点，完整见 [02.5 §8](./02.5-张量并行实现详解.md)）

TP 把注意力/MLP 的矩阵乘切开了，但 **LayerNorm / Dropout / 残差相加**这些「逐 token」算子 TP 不切——每个 TP rank 都留一份全量 `[s,b,h]`，浪费。**SP 就把这些区间也沿序列切成 `[s/TP,b,h]`**：

- **切的范围**：只有 LN/Dropout/残差这些「TP 会留全量」的窄区间；矩阵乘/注意力区仍是 TP 在切。
- **进出转换**（`megatron/core/tensor_parallel/mappings.py`）：
  - 进 TP 矩阵乘区：`_GatherFromSequenceParallelRegion`（`:296`，All-Gather 沿序列 → 拼回全 `s`）
  - 出 TP 矩阵乘区：`_ReduceScatterToSequenceParallelRegion`（`:351`，Reduce-Scatter → 跨 TP 求和 + 切回 `s/TP`）
  - 效果：把纯 TP 的一次 **All-Reduce 拆成 AG+RS 两半，通信总量不变**（这正是 EP 篇 §3 Remark 里 ④⑥ 那对 AG/RS 的同款动作）。
- **注意力看到完整序列**：SP 一进注意力就 all-gather 拼回全序列，所以 **SP 从不碰「token 跨卡互相看」这个难题**——注意力照常在全序列上算（TP 分的是**头**，不是序列）。
- **绑死 TP**：SP 度 = TP 度、复用 TP 组（`sequence_parallel` 是 TP 上的一个开关），不能脱离 TP 单独存在。

### 5.2 一句话对照

- **SP**：切了**再拼回来**（省显存），注意力见全序列、**无 KV 通信**。
- **CP**：切了**就不拼**、用 **KV 通信**硬扛注意力依赖（扩长度）。

| | **SP 序列并行** | **CP 上下文并行** |
|---|---|---|
| 用哪个组 | 复用 **TP 组**（SP 度 = TP 度） | **专属 CP 组** `get_context_parallel_group()`，`DP×TP×PP×CP×EP` 里独立一维 |
| 切序列的范围 | 只切 LN/Dropout/残差等窄区间 | **整层都切**，含注意力 |
| 每卡持有 | `s/TP`（仅 SP 区间，注意力处拼回全 `s`） | `s/CP`（全程） |
| **注意力怎么办** ★ | all-gather 拼回**全序列**本地算，**无跨卡 KV** | 序列始终切着，跨 token 依赖靠 **KV 交换**（Ring/Ulysses/AG-KV） |
| 通信 | AG↔RS 成对（替换 TP 的 all-reduce，量不变） | 注意力处额外 KV 交换（P2P/A2A） |
| 目的 | 省激活显存（TP 的补丁） | 扩序列长度（独立一把刀） |

### 5.3 多机多卡对比例子

16 卡 = 2 节点 × 8 卡。拿同一段 8-token 序列 `t0..t7`，看两种配置在**注意力那一步**的差别（为看清，聚焦相邻 2 卡 `{G0,G1}`）：

```
配置 A ── TP=2 + SP（{G0,G1} 是 TP 组，机内 NVLink）
  LN/Dropout 区:  G0=[t0 t1 t2 t3]   G1=[t4 t5 t6 t7]     ← 沿序列切，省显存
  进注意力 ─ All-Gather(TP组) ─►  G0=[t0..t7]  G1=[t0..t7] ← 拼回全序列
  算注意力:       G0 算「头 0..h/2」  G1 算「头 h/2..h」    ← TP 切的是头，各自看全序列
  出注意力 ─ Reduce-Scatter(TP组) ─► G0=[t0..t3] G1=[t4..t7]← 切回半段
  ★ 注意力期间 KV 跨卡通信：无（全序列已在本地）

配置 B ── CP=2（{G0,G1} 是 CP 组）
  全程:           G0=[t0 t1 t2 t3]   G1=[t4 t5 t6 t7]     ← 一直切着，注意力也不拼
  算注意力(因果):  G0 的 q(t0..t3) 只需 t0..t3 的 KV（本地全有）
                  G1 的 q(t4..t7) 需要 t0..t7 的 KV → 向 G0 要 t0..t3 的 KV
  ─ Ring/P2P 交换 KV ─►  G1 收到 G0 的 K/V，用 online-softmax 累加
  ★ 注意力期间 KV 跨卡通信：有（这就是 CP 的本质，§3）
```

**放大到多机的关键差别**：

- **SP 绑死 TP**：它的 AG/RS 每层多次、量大，只能关在 **机内 NVLink**（跟 TP 一起，≤8 卡不跨节点）。
- **CP 有独立组、可跨节点**：KV 交换能用 P2P **流水重叠**藏进计算（Ring），所以 CP 组**可以跨机**扩序列长度；这也是为什么 CP 常和 DP 合成 `dp-cp` 组（§2）。

> 所以 SP 和 CP 是**正交、可叠加**的：一个模型完全可以「TP=8+SP」关在机内、再套「CP=4」跨 4 个节点——SP 在机内省 LN/Dropout 的显存，CP 在机间把超长序列切开。长序列工具箱里的 2D CP 正是这类叠加（[02.9 §2.2](./02.9-进阶专题.md)）。

## 6. CP 小结

- CP 沿**序列维**切，专为**超长上下文**；MLP/LN 无影响，唯一通信在**注意力的 KV 交换**。
- 四种 `cp_comm_type`：`p2p`(ring) / `all_gather` / `a2a`(Ulysses) / `a2a+p2p`，默认推荐 ring。
- Ring Attention 靠 **online-softmax 分块累加** + **异步 P2P 重叠**实现。
- **SP ≠ CP**（§5）：SP 是 TP 的省显存附庸（注意力 all-gather 拼回全序列、无 KV 通信、绑机内 NVLink），CP 是独立并行维（注意力切着算、靠 KV 交换、可跨节点）；两者正交可叠加。
- CP 与其余四种并行的横向对比、放置直觉，见 [02.8 · 专家并行 §6 五种并行通信特征总表](./02.8-专家并行.md)。
