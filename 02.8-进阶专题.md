# 02.8 · 进阶专题：Muon 分布式化 / 长序列训练 / FSDP vs TP+PP

## 1. 专题①：Muon 优化器的分布式化难题

Muon 的单卡数学（动量矩阵做 Newton-Schulz 正交化 `O = NS(M) ≈ UVᵀ`）见 [02.2 §3](./02.2-优化器数学原理与对比.md)。这里只讲一个它带来的新麻烦：**Muon 和现有并行"天然打架"。**

### 1.1 根本矛盾：正交化要"完整矩阵"，并行却把矩阵切碎了

Newton-Schulz 迭代是**在整个二维权重矩阵 `W` 上做的矩阵乘**——必须拿到**完整矩阵**才能算对。可分布式训练恰恰到处在切矩阵：

| 并行 | 把矩阵切成了什么 | 冲突 |
|------|-----------------|------|
| **ZeRO / FSDP** | 梯度 Reduce-Scatter 后每卡只有 `1/N` 段，参数也可能分片 | 在分片上做 NS ≠ 在完整矩阵上做 NS |
| **TP** | 权重按行/列切，每卡只有子矩阵 | 同上，子矩阵正交化结果不对 |
| **MoE** | 每个专家是一个独立矩阵，散在不同卡 | 需明确"哪张卡负责算哪个专家的 Muon 更新" |

此外还有一层：NS 本身是几个矩阵乘，有额外算力，**怎么把这份算力也并行掉**（是"聚合到单卡算"还是"分布式矩阵乘"）也是权衡。

### 1.2 主流做法（Moonlight / Megatron 实践）

- **先聚合再更新**：沿分片维把某个矩阵的梯度 **All-Gather 成完整块** → 在一张卡上跑 NS → 结果再 scatter 回去。Moonlight 的《Muon is Scalable》正是在 ZeRO-1 上这么做的。
- **参数级并行**：把"不同矩阵的 NS 计算"分给不同 rank 并行做，各算各的、再 Broadcast 回去，把额外算力摊薄。
- **TP/EP**：沿被切维度先 gather 再正交化；MoE 把专家矩阵按 rank 分派，谁持有谁算。

### 1.3 权衡一句话

**Muon 的收益 vs 为"拼出完整矩阵"付出的额外通信 + 算力。** 切得越细、矩阵越大，这份"组装成本"越高——这也是 Muon 大规模落地时的核心工程问题。

## 2. 专题②：长序列训练的显存工具箱

当序列长到 128K、1M，[02.1 §2.2](./02.1-显存、激活值与重计算.md) 说的"激活 ∝ b·s·h·L"会爆，而且冒出三个新的显存爆点，每个都有对应招法。三者**正交，可叠加**。

### 2.1 爆点① LM Head 的巨型 logits → Cut Cross-Entropy

最后的语言模型头输出 logits 形状 `[b·s, vocab]`，而 `vocab` 常达 10万–20万。长序列下这块 logits **比模型本身还大**。

**招法**：不物化完整 logits，**分块在片上（on-chip）算 loss 和梯度**，只保留每行的 logsumexp 统计量。logits 显存从 `O(s·vocab)` 降到 ~`O(1)`。

> 这和 [02.1 §5.4 FlashAttention](./02.1-显存、激活值与重计算.md) 是同一套"分块计算 + 只留统计量"的思路，换了个战场（从注意力矩阵换到 logits）。

### 2.2 爆点② 注意力激活 ∝ 序列长 → 2D 上下文并行

单一 CP 方式各有短板：Ulysses(A2A) 受注意力头数约束，Ring(P2P) 沿序列扩展但跳数多。

**招法**：把 Megatron 的两种 CP **组合成 2D CP**——**Ulysses 切注意力头 + Ring 沿序列延展**，再叠加 **Zigzag 因果均衡**（见 [02.7 §因果负载均衡](./02.7-上下文并行与专家并行.md)），支撑更长序列。

### 2.3 爆点③ 模型态 + 残差激活 → FSDP + 重计算

**招法**：**FSDP** 把 16P 模型态切成 `16P/N`（卡越多越省，见 [02.4 §2.6](./02.4-并行组构建与通信详解.md)）+ **重计算**省激活（[02.1 §5](./02.1-显存、激活值与重计算.md)）。这是长序列下最省显存的模型态/激活组合拳。

### 2.4 小结

| 爆点 | 招法 | 效果 |
|------|------|------|
| logits `[b·s, vocab]` | **Cut Cross-Entropy** | `O(s·vocab)` → ~`O(1)` |
| 注意力激活 ∝ s | **2D CP**（Ulysses+Ring+Zigzag） | 沿头 + 沿序列双向切 |
| 模型态 + 激活 | **FSDP + 重计算** | 模型态 `/N`、激活重算 |

三者可**同时开**：Cut-CE 管 logits、CP 管注意力、FSDP+重计算 管模型态与残差激活。

## 3. 专题③：FSDP vs TP+PP 怎么选

主流大模型训练通常在 **FSDP** 和 **TP+PP** 之间二选一（**EP 与两者正交**，可叠加在任一之上）。

### 3.1 为什么不总是 TP+PP

显存吃紧时，TP+PP 会逼你用**很小的 micro-batch** → 单卡每次算的量太少、**算术强度（arithmetic intensity）低** → **MFU（算力利用率）低**。换成 FSDP：省下的显存拿去**放大 micro-batch** → 抬高算术强度 → MFU 上来了。

### 3.2 但 FSDP 有规模天花板

FSDP 的通信是**全局域**的（整个 DP 域一起 All-Gather / Reduce-Scatter 参数）。**上千卡以后，很容易被网络带宽卡住**。关键指标是 FSDP 的**通信隐藏度**（通信能被计算掩盖多少）：

| 通信隐藏度 | 建议 |
|-----------|------|
| **≥ 80%**（通信基本藏进计算） | 用 **FSDP**，吃 MFU 红利 |
| **< 80%**（通信裸露、卡带宽） | 用 **TP+PP**：重通信留在**机内 NVLink**，跨节点只走轻量的 PP |

### 3.3 一句话选型

**小规模 / 被显存逼到 micro-batch 太小 → FSDP 提 MFU；超大规模 / 通信藏不住 → TP+PP（把重通信关在机内）。**

---

## 4. 全篇小结

- **Muon 分布式化**：正交化要完整矩阵，而 ZeRO/FSDP/TP/MoE 都在切矩阵 → 需"先聚合再 NS / 参数级并行"，代价是额外通信与算力。
- **长序列三招**（正交可叠加）：Cut Cross-Entropy 削 logits、2D CP（Ulysses+Ring+Zigzag）削注意力、FSDP+重计算 削模型态与激活。
- **FSDP vs TP+PP**：以"通信隐藏度 80%"为界——藏得住用 FSDP 提 MFU，藏不住用 TP+PP 把重通信关进机内 NVLink；EP 与两者正交。
