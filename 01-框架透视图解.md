# 01 · 框架透视图解

本篇建立 Megatron-LM 的全局视图：双层架构定位、目录结构、分层依赖关系总图，以及端到端执行链路鸟瞰。后续各篇是对这张总图中某个方块的放大。

---

## 1. 双层架构定位

Megatron-LM 仓库实际上交付**两个产物**，二者同处一个仓库但定位不同：

```mermaid
flowchart TB
    subgraph REPO["Megatron-LM 仓库"]
        subgraph LM["Megatron-LM 训练参考实现（应用层）"]
            direction LR
            ENTRY["根目录入口脚本<br/>pretrain_gpt/hybrid/mamba/vlm.py, train_rl.py"]
            TRAIN["megatron/training/<br/>参数解析 · 训练循环 · 检查点 · 日志"]
        end
        subgraph CORE["Megatron Core（库层，发布为 PyPI 包 megatron-core）"]
            direction LR
            MODELS["models 模型"]
            TRANSF["transformer 构件"]
            PARA["并行化 TP/PP/CP/EP"]
            DIST["distributed / optimizer"]
            CKPT["dist_checkpointing"]
            INFER["inference"]
        end
    end
    LM --> CORE
    CORE --> TORCH["PyTorch ≥ 2.6 + Transformer Engine / CUDA Kernels"]
```

| 维度 | Megatron Core | Megatron-LM 训练实现 |
|------|---------------|----------------------|
| 路径 | `megatron/core/` | `megatron/training/` + 根目录脚本 |
| 定位 | 可组合的底层库 | 开箱即用的训练框架 |
| 受众 | 框架开发者 / ML 工程师 | 研究者 / 快速实验 |
| 发布 | PyPI `megatron-core` | 随仓库提供，不单独发布 |
| 稳定性 | 有 API 向后兼容约束 | 偏脚本，灵活演进 |

> 关键认知：`megatron/training/` **依赖** `megatron/core/`，反之不成立。Core 不感知具体训练脚本，这是整个仓库最重要的分层边界。

## 3. 分层依赖关系总图（核心）

整个代码仓最关键的就是这张**自上而下、上层依赖下层**的分层图。每一层只调用同层或更底层的能力。

```mermaid
flowchart TD
    L0["L0 入口层：pretrain_gpt.py / train_rl.py ...<br/>定义 get_batch / forward_step，调用 pretrain()"]
    L1["L1 训练框架层 megatron/training/<br/>pretrain() · train() · 参数系统 · 检查点 · 日志"]
    L2["L2 模型层 megatron/core/models/<br/>GPTModel · MambaModel · HybridModel · 多模态"]
    L3["L3 Transformer 构件层 megatron/core/transformer/<br/>TransformerBlock · TransformerLayer · Attention · MLP · MoE"]
    L4["L4 并行原语层 tensor_parallel / pipeline_parallel<br/>列/行并行线性层 · 1F1B 调度 · P2P 通信"]
    L5["L5 分布式基础层 parallel_state(mpu) / distributed / optimizer<br/>并行组构造 · DDP/FSDP · 分布式优化器"]
    L6["L6 运行时底座：PyTorch · Transformer Engine · NCCL · CUDA Kernels"]

    L0 --> L1 --> L2 --> L3 --> L4 --> L5 --> L6
    L1 -.横切.-> CK["dist_checkpointing"]
    L1 -.横切.-> DS["datasets / tokenizers"]
    L2 -.复用.-> CK
    L5 -.被各层调用.-> L4
```

### 横切关注点（被多层共享）

这些模块不在主调用链上，而是被多层横向调用：

- **`parallel_state.py`（mpu）**：全局并行组状态，几乎被所有层查询「我属于哪个 TP/PP/DP 组」。
- **`dist_checkpointing/`**：模型层产生 `ShardedStateDict`，训练层负责落盘/加载。
- **`datasets/` + `tokenizers/`**：被训练层与入口层共同使用。
- **`config` 体系**：`ModelParallelConfig` → `TransformerConfig` 配置对象贯穿模型与并行层。

## 4. 配置对象继承链

Megatron 用**配置对象**驱动模型构建，理解这条继承链是读懂代码的钥匙：

```mermaid
flowchart LR
    MPC["ModelParallelConfig<br/>(model_parallel_config.py)<br/>TP/PP/CP/EP 尺寸、通信重叠开关"]
    TC["TransformerConfig<br/>(transformer/transformer_config.py)<br/>层数、隐藏维、注意力头、MoE 配置"]
    MPC --> TC
    TC --> GPT["GPTModel 等具体模型"]
    CLI["命令行参数 (arguments.py)"] -->|core_transformer_config_from_args| TC
```

- `ModelParallelConfig`：定义并行维度与通信策略。
- `TransformerConfig`（继承前者）：追加 Transformer 结构超参，是模型构建的统一入口配置。
- 训练侧由 `core_transformer_config_from_args()` 把 CLI 参数翻译成 `TransformerConfig`。

## 5. 端到端执行链路鸟瞰

以最常见的 GPT 预训练为例，一次完整运行的宏观链路：

```mermaid
sequenceDiagram
    participant U as 用户/torchrun
    participant E as pretrain_gpt.py (L0)
    participant T as training.pretrain() (L1)
    participant MPU as parallel_state (L5)
    participant M as GPTModel (L2~L4)
    participant D as DataLoader (datasets)
    participant O as DistributedOptimizer (L5)

    U->>E: torchrun ... pretrain_gpt.py [args]
    E->>T: pretrain(cfg, dataset_provider, model_provider, forward_step)
    T->>MPU: initialize_megatron() 建立 TP/PP/DP/CP/EP 进程组
    T->>M: setup_model_and_optimizer() 构建模型并包 DDP
    T->>O: 创建分布式优化器 + LR 调度
    T->>D: build_train_valid_test_data_iterators()
    loop 每个训练迭代
        T->>M: train_step() → forward_backward_func (1F1B 流水线)
        M->>M: forward_step 计算 loss
        M->>O: backward 累积梯度到梯度桶
        O->>O: 梯度规约/裁剪 + optimizer.step()
        T->>T: training_log() 记录指标
        T->>T: 按间隔 save_checkpoint() (分布式检查点)
    end
```

链路中每个角色对应一个子系统，后续文档逐一展开：

| `initialize_megatron` / 并行组 | [02 并行化子系统](./02-并行化子系统.md) |
| `GPTModel` / forward | [03 Transformer 与模型](./03-Transformer与模型子系统.md) |
| DDP / 优化器 / 梯度 | [04 分布式训练与优化器](./04-分布式训练与优化器.md) |

## 6. 关键依赖与环境

来自 `pyproject.toml`：

- **Python**：`>= 3.12`（0.17+ 起放弃 3.10）
- **核心依赖**：`torch >= 2.6.0`、`numpy`、`packaging`
- **训练扩展 `[training]`**：`transformers`、`wandb`、`sentencepiece`、`tiktoken`、`accelerate`、`omegaconf`
- **开发扩展 `[dev]`**：`flashinfer`、`megatron-energon`、`nvidia-modelopt`、`emerging_optimizers`、`tensorstore`、`multi-storage-client` 等
- **隐性强依赖**：NVIDIA **Transformer Engine**（FP8/融合 kernel）、**NCCL**（集合通信）、CUDA

安装：

```bash
cd Megatron-LM
uv pip install -e .                 # 仅 megatron-core
uv pip install -e ".[training]"     # 完整训练栈
```

## 7. 小结

- 仓库 = **Core 库** + **训练参考实现**两层，边界清晰，方向单向。
- 代码组织遵循严格的**六层依赖**：入口 → 训练 → 模型 → Transformer → 并行原语 → 分布式底座。
- `parallel_state`、`dist_checkpointing`、`config` 体系是贯穿各层的**横切关注点**。
- 一切运行最终落到 PyTorch + Transformer Engine + NCCL 之上。
