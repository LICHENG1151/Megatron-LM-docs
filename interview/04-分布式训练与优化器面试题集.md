## 显存与激活（轴 B）

### Q1 🟢〔轴B〕混合精度 + Adam 训练时，**一个参数**占多少字节显存？拆开算。

**答案**：约 **16 字节 / 参数**（不含激活）：

| 项 | 精度 | 字节 |
|----|------|:--:|
| 参数（计算用） | fp16/bf16 | 2 |
| 梯度 | fp16/bf16 | 2 |
| 参数副本 master weight | fp32 | 4 |
| Adam 一阶动量 `m` | fp32 | 4 |
| Adam 二阶动量 `v` | fp32 | 4 |

合计 16 B/param → 7B 模型光模型态就 ~112 GB，单卡装不下，这是所有并行/切分的根本动机。SGD 无 `m/v` 则为 8 B/param。

**关联**：这笔账是 ZeRO / 分布式优化器（把后 12 B 切成 1/N）与 Q4 的出发点。详见 [02.1](../02.1-显存、激活值与重计算.md)、[02.2](../02.2-优化器数学原理与对比.md)。

### Q2 🟡〔轴B〕激活显存为什么随 `batch × seq × layers` 增长？重计算（activation checkpointing）拿什么换什么？

**答案**

- 前向每层都要**保存中间激活**供反向用，量级 ∝ `b · s · h · L`（层数 L 线性叠加），长序列时 attention 还有 `O(s²)` 项。大模型训练里激活常是**比模型态更大的显存大头**。
- **重计算**：前向**只存每层入口**（或每若干层），反向时**重新前向算一遍**得到中间激活。用**多一次前向的算力**换**激活显存大幅下降**（典型省到 `O(√L)` 或按选择性重算）。
- FlashAttention 是同一思想在 attention 内的特例：不存 `s×s` 注意力矩阵，反向重算，激活 `O(s²)→O(s)`（[02.1.1](../02.1.1-FlashAttention.md)）。

**伪代码**：

```python
# 普通：存所有中间激活
def block(x): h = f(x); return g(h)          # h 被保存，反向直接用

# 重计算：入口存 x，反向重算 f(x)
def block_ckpt(x):
    return checkpoint(lambda t: g(f(t)), x)  # 前向不存 h；反向重跑 f→h 再回传
```

### Q3 🟡〔轴B〕混合精度训练为什么要额外保留一份 **fp32 master weight**？

**答案**

参数更新量 `lr · m̂/(√v̂+ε)` 常常**远小于**参数本身。若直接在 fp16 上更新，小增量在 fp16 有效位数下被**舍入吞掉**（下溢），训练停滞。做法：前向/反向用 fp16 省显存和带宽，但**优化器在 fp32 master weight 上更新**，再 cast 回 fp16 用于下一次前向。这是 Q1 里那 4 B fp32 副本的来由。

配套还有 **loss scaling**（见 Q10）解决 fp16 梯度**下溢**。

### Q4 🟡〔轴B〕分布式优化器 / ZeRO-1 把优化器状态切成 1/N，它把 DP 的 all-reduce 换成了什么？通信量变了吗？

**答案**

- 标准 DP：反向后对梯度做 **all-reduce**，每卡持完整梯度、各自更新完整优化器状态（Q1 的 12 B 每卡都存一份，冗余）。
- 分布式优化器（≈ZeRO-1）：把梯度 **reduce-scatter**（每卡只拿到自己负责的那 1/N 段的全和），各卡只更新 1/N 的参数与优化器状态，更新完再 **all-gather** 把参数补回。
- 因为 `AllReduce = ReduceScatter + AllGather`，**总通信量不变（≈2S）**，却把优化器状态显存从每卡 12 B 降到 12 B/N。

**Megatron 实现**（`megatron/core/distributed/param_and_grad_buffer.py`）：梯度桶用 reduce-scatter 而非 all-reduce：

```python
# 分布式优化器开启时，梯度同步走 reduce-scatter（每卡只留自己那片的全和）
dist_reduce_scatter_func = torch.distributed.reduce_scatter_tensor       # :40
if self.ddp_config.use_distributed_optimizer:
    dist_reduce_scatter_func(local_shard, bucket_grad, group=dp_group)   # 反向末尾
# 优化器 step 后再 all_gather 把更新过的 param 补回各卡（distrib_optimizer.py:1322）
```

**关联**：这与 SP 把 TP 的 all-reduce 拆成 RS+AG 是**同一个恒等式的两个应用**（一个切优化器态、一个切激活，见 [02·Q8](./02-并行化子系统面试题集.md)）。

### Q5 🔴〔轴B〕ZeRO-1 / 2 / 3 分别切什么？显存与通信如何权衡？和 TP 的取舍是什么？

**答案**

| 级别 | 切什么 | 显存 | 通信 |
|------|--------|------|------|
| ZeRO-1 | 优化器状态（`m/v`+master） | 省最多的那 12 B/param → 1/N | ≈ DP（RS+AG，总量不变） |
| ZeRO-2 | + 梯度 | 再省梯度 | ≈ DP |
| ZeRO-3 | + 参数本身 | 参数也切 1/N，省到极致 | **多**：前向/反向需按需 all-gather 参数，通信量上升 |

**取舍**：
- ZeRO-1/2 几乎"免费省显存"（通信量不变），一般优先开。
- ZeRO-3 省最多但通信显著增加，且每次用参数都要 gather。
- 与 **TP** 的区别：TP 切**单层内的权重与激活**、通信在**层内关键路径**（每层多次 all-reduce），适合节点内高带宽；ZeRO 切**跨 DP 副本的冗余状态**、通信在**反向末尾/参数访问**，更适合带宽没那么高的场景。大规模常**组合**：TP(节点内) + ZeRO-1(DP 维) + PP(跨节点)。详见 [02.3.1](../02.3.1-FSDP.md)。

### Q6 🟡〔轴B〕梯度累积（gradient accumulation）如何在不增显存的情况下放大等效 batch？和 DP 是什么关系？

**答案**

- 把一个大 batch 拆成 `k` 个 micro-batch，**逐个前向+反向、梯度累加**到同一 buffer，累计 `k` 次后再做一次 optimizer step。等效 batch = `micro_batch × k × dp_size`，但**峰值激活显存只按一个 micro-batch 计**。
- 代价：`k` 次前反向串行，吞吐不变但 step 变长；与 DP 正交——DP 在**空间**上并行放大 batch，梯度累积在**时间**上放大。
- 在 PP 里，micro-batch 正是 1F1B 流水的调度单位（见 [02·Q14](./02-并行化子系统面试题集.md)）。

**伪代码**：

```python
optimizer.zero_grad()
for i in range(k):                       # k = 梯度累积步数
    loss = model(next(micro_batch)) / k  # 除以 k 等价于对 k 个 micro-batch 求平均
    loss.backward()                      # 梯度累加进 .grad，不清零
# k 步后：DP all-reduce/reduce-scatter 梯度 → optimizer.step()
sync_gradients(); optimizer.step()
```

### Q7 🟡〔轴C〕DP 的梯度同步如何与反向计算**重叠**？为什么要分桶（bucket）？

**答案**

- 反向是**逐层从后往前**算的，某层梯度一算完就可以开始同步，不必等整个反向结束。Megatron 把参数分成若干 **bucket**，某个 bucket 的梯度**一凑齐就异步发起 reduce-scatter/all-reduce**，与更前面层的反向计算**重叠**，把通信藏进计算里。
- 分桶原因：① 太细则 kernel 太多、NCCL 启动开销大；② 太粗则重叠窗口小、要等很久才凑齐一桶。所以按一个合理的 `bucket_size` 折中。

**Megatron 实现**（`megatron/core/distributed/`）：

```python
# distributed_data_parallel.py：按 bucket 异步 reduce，overlap_grad_reduce 且 pp_rank==0 时开启
# bucket_size 默认随 dp_size 放大以保证 NCCL 分片效率
ddp_config.bucket_size = max(40_000_000, 1_000_000 * dp_group.size())     # :69
# param_and_grad_buffer.py：每桶 grad ready → 异步 reduce-scatter（分布式优化器）/all-reduce
```

### Q8 🔴〔轴C〕为什么 TP 通常放在**单节点内**、PP/DP 可以跨节点？

**答案**

按**通信频率 × 是否在关键路径 × 带宽需求**排：

- **TP**：每层前向 2 次、反向 2 次 all-reduce，**在关键路径上**（下一步计算等它），通信量大且频繁 → 必须高带宽（**NVLink/NVSwitch，节点内 ~数百 GB/s**）。跨节点走 IB（~几十 GB/s）会严重拖慢。
- **PP**：只在 stage 边界 P2P 传激活，**通信量小、频率低** → 跨节点（IB）也能接受。
- **DP**：反向末尾同步一次梯度，且能与反向**重叠**（Q7）→ 跨节点可接受。

所以经验放置：**TP ≤ 单节点 GPU 数、走 NVLink；PP 跨节点；DP 最外层补满**。这也是"TP 不宜开太大"的根因（见 [06·Q4](./06-训练框架Harness面试题集.md)）。

## 优化器数值（轴 D）

### Q9 🟡〔轴D〕分布式下**全局梯度裁剪**（clip by global norm）怎么算？为什么不能各卡各裁各的？

**答案**

全局范数是**所有参数梯度**拼在一起的 L2 范数：`total_norm = sqrt(Σ_all g_i²)`。但梯度被切在 TP/PP/DP 各卡上，**任何单卡都只有一部分**，所以必须**跨模型并行组 all-reduce 求和**局部平方和，得到真正的全局范数，再统一乘裁剪系数。各卡各裁会得到不同的 `total_norm`、破坏一致性。

**Megatron 实现**（`megatron/core/optimizer/clip_grads.py`）：

```python
# 各卡先算局部平方和，再跨并行组 all-reduce SUM 得全局范数         :124-140
total_norm = sum(grad_norm ** norm_type for grad in grads_for_norm)  # 局部
torch.distributed.all_reduce(total_norm, op=ReduceOp.SUM, group=data_parallel_group)     # :133
torch.distributed.all_reduce(total_norm, op=ReduceOp.SUM, group=grad_stats_parallel_group) # :136 跨TP/PP
total_norm = total_norm ** (1.0 / norm_type)
# inf-norm 走 MAX 而非 SUM（:99-103）
clip_coef = max_norm / (total_norm + 1e-6)
if clip_coef < 1.0: grads.mul_(clip_coef)
```

### Q10 🟡〔轴D〕混合精度里的 **loss scaling** 为什么必要？动态调整机制是怎样的？

**答案**

- **为什么**：fp16 能表示的最小正数有限，**小梯度会下溢成 0**，训练学不动。做法：反向前把 loss **乘一个大 scale `S`**，梯度同比放大避免下溢；optimizer step 前再**除回 `S`**（unscale）。
- **动态调整**（`DynamicGradScaler`）：无 inf/nan 时，连续 `growth_interval` 步后把 `S` **乘 growth_factor** 变大（尽量用大 scale 保精度）；出现 inf/nan 时，**跳过这一步更新**、把 `S` **乘 backoff_factor** 变小，并有 `hysteresis` 容忍连续几次才真正降。

**Megatron 实现**（`megatron/core/optimizer/grad_scaler.py` `DynamicGradScaler.update`）：

```python
def update(self, found_inf):
    if found_inf:
        self._growth_tracker = 0
        self._hysteresis_tracker -= 1
        if self._hysteresis_tracker <= 0:                              # :142
            self._scale = torch.max(self._scale * self.backoff_factor, self.min_scale)  # 降 scale
    else:
        self._growth_tracker += 1
        if self._growth_tracker >= self.growth_interval:               # 连续无 nan
            self._hysteresis_tracker = self.hysteresis
            self._scale = self._scale * self.growth_factor             # 升 scale
```

> bf16 动态范围大、通常**不需要** loss scaling，这也是很多训练用 bf16 的原因之一。
