## 训练主循环（轴 D）

### Q1 🟢〔轴D〕描述一次完整训练 step 的数据流（从 batch 到参数更新）。

**答案**

```
1. 取 batch → tokenize → 各并行组内广播/切分输入
2. 前向：embedding(词表并行) → N×TransformerLayer(TP/SP/PP) → 输出层
3. 算 loss：并行交叉熵(词表并行，不聚合全量 logits)
4. 反向：逐层回传，产生梯度（TP 层内、PP 层间 P2P）
5. 梯度同步：DP all-reduce 或 reduce-scatter(分布式优化器)，与反向重叠
6. 梯度处理：unscale(loss scaling) → 全局梯度裁剪 → 检查 inf/nan
7. optimizer.step()：fp32 master weight 上更新 → cast 回 fp16
8. 更新 loss scale、lr scheduler，zero_grad
```

考察点：能否把**并行维度**（谁在第 2/4/5 步通信）和**混合精度步骤**（第 6/7 步）串起来。梯度累积则在 2–4 步重复 k 次后才进 5–7（见 [04·Q6](./04-分布式训练与优化器面试题集.md)）；第 6 步的梯度裁剪 / loss scaling 细节见 [04·Q9](./04-分布式训练与优化器面试题集.md)、[04·Q10](./04-分布式训练与优化器面试题集.md)。

## 工程与排障（轴 E）

> `CUDA_DEVICE_MAX_CONNECTIONS=1` 的必要性见 [02·Q9](./02-并行化子系统面试题集.md)。

### Q2 🟡〔轴E〕loss 正常但 **MFU / 吞吐很低**，你按什么顺序定位？

**答案**（从"看数据"到"下结论"的路径，而非乱猜）：

1. **先测不猜**：用 `torch.profiler` / Nsight 抓一段 timeline，看 **计算 vs 通信 vs 空泡** 占比。
2. **通信是否重叠**：DP 梯度同步、TP/SP 通信有没有和计算 overlap？没重叠先修 overlap（`overlap_grad_reduce`、`CUDA_DEVICE_MAX_CONNECTIONS`）。
3. **流水线气泡**：PP 是否 bubble 过大？micro-batch 数够不够、要不要交错式。
4. **TP 是否过大**：TP 通信在关键路径，TP 越大占比越高（见 Q4）。
5. **数据/IO**：dataloader 是否喂不上、CPU 预处理是否瓶颈（GPU 等数据）。
6. **重计算过度**：是否为省显存开了过多重算，拿吞吐换了显存。

好答案的共性：**先 profiler 定位、再按占比对症**，而不是直接"加卡/调 batch"。

### Q3 🟡〔轴E〕训练 **OOM**，把可用手段**按代价从小到大**排出来。

**答案**

| 手段 | 省什么 | 代价 |
|------|--------|------|
| 减小 micro-batch + 增梯度累积 | 激活 | 几乎无（step 变长），首选 |
| 激活重计算（选择性→全量） | 激活（大头） | 多一次前向算力 |
| 开启序列并行 SP | 激活（LayerNorm/Dropout 区） | 通信量不变，几乎免费 |
| 分布式优化器 / ZeRO-1 | 优化器状态 12→12/N | 通信量不变，几乎免费 |
| 提高 TP | 层内权重+激活 | TP 通信在关键路径，吞吐下降 |
| 提高 PP | 跨层 | 引入气泡 |
| ZeRO-3 / offload | 参数/一切 | 通信大增 / PCIe 慢 |

考察点：能否**区分"几乎免费"（SP、ZeRO-1、重计算）与"拿吞吐换显存"（提 TP/PP、offload）**，并有先后次序，而不是一上来就加并行度。相关机制见 [04·Q2/Q4/Q5](./04-分布式训练与优化器面试题集.md)、[02·Q8 SP](./02-并行化子系统面试题集.md)。

### Q4 🔴〔轴E〕TP 从 2 调到 8，吞吐不升反降，为什么？什么时候该把并行度让给 PP？

**答案**

- **为什么降**：TP 每层前反向共 4 次 all-reduce，**在关键路径上**。TP 越大：① 每次 all-reduce 参与卡更多、单卡算的 GEMM 更小（计算/通信比变差）；② 一旦 TP 组**跨出 NVLink 域**（如 TP=8 但单节点只有 4 卡高带宽互联），通信走慢速链路，直接拖垮。
- **何时转 PP**：当 TP 已占满单节点高带宽域、再加 TP 就要跨节点时，应把扩展让给 **PP（跨节点、通信量小）** 或 **DP**。经验：**TP ≤ 单节点 NVLink GPU 数**，超出规模用 PP×DP 承接。
- 配合看：用 profiler 确认瓶颈确实是 TP 通信占比上升（呼应 Q2 与 [04·Q8 带宽放置](./04-分布式训练与优化器面试题集.md)）。

### Q5 🔴〔轴E〕多机训练 loss 突然变 **NaN**，排查路径？

**答案**（分层排除）：

1. **是不是溢出**：看 loss scale 是否在反复 backoff、grad 是否 inf。fp16 下先怀疑 loss scaling 配置（初始 scale 过大/过小，见 [04·Q10](./04-分布式训练与优化器面试题集.md)）。
2. **数据**：是否有坏样本（超长、全 pad、异常 token）；定位到具体 step/batch 复现。
3. **数值**：lr 是否过大、warmup 是否太短、某些 norm/softmax 是否缺数值稳定项。
4. **通信/一致性**：多机某 rank 是否梯度不同步（如裁剪范数没跨组 all-reduce，见 [04·Q9](./04-分布式训练与优化器面试题集.md)），导致参数逐渐发散→溢出。
5. **硬件**：个别节点 ECC/网络错误。用"固定 seed + 关重算 + 缩小到单机"逐步二分。

考察点：是否有**系统性二分思路**（改一个变量、能复现），而非"重启试试"。
