## 一、基础概念题

### Q1 🟢 `ColumnParallelLinear` 和 `RowParallelLinear` 的权重分别沿哪个维度切分？为什么这样切？

**答案**

设线性层 `Y = XAᵀ`（PyTorch `F.linear` 存的是转置权重）。

- **ColumnParallelLinear（列切）**：权重 `A` 按**输出维度（列）**切成 `[A₁, …, Aₚ]`。存储 shape 为 `[output_size_per_partition, input_size]`，`partition_dim=0`。
  - `layers.py:870` `self.output_size_per_partition = divide(output_size, world_size)`
  - 每卡算 `Yᵢ = X·Aᵢᵀ`，得输出的一个列块 `[s, b, output_size/p]`。
- **RowParallelLinear（行切）**：权重按**输入维度（行）**切，存储 shape `[output_size, input_size_per_partition]`，`partition_dim=1`。
  - `layers.py:1221` `self.input_size_per_partition = divide(input_size, world_size)`
  - 输入已按最后一维切成 `Xᵢ`，每卡算部分和 `Yᵢ = Xᵢ·Aᵢᵀ`，`Y = ΣYᵢ`。

**为什么**：列切的输出块正好是行切需要的输入分片，**「列切 → 行切」直接对接、中间不通信**——这是 TP 设计的核心。

**Megatron 实现**（`megatron/core/tensor_parallel/layers.py`）：

```python
# ColumnParallelLinear.__init__  —— 按输出维(dim=0)切
# 注：F.linear 算 XA^T，故存的是转置权重，切输出维 = 切存储的 dim 0
self.output_size_per_partition = divide(output_size, world_size)          # :870
self.weight = Parameter(torch.empty(
    self.output_size_per_partition, self.input_size,                      # [out/p, in]
    device=torch.cuda.current_device(), dtype=config.params_dtype))
_initialize_affine_weight_gpu(self.weight, init_method, partition_dim=0, ...)  # :913

# RowParallelLinear.__init__  —— 按输入维(dim=1)切
self.input_size_per_partition = divide(input_size, world_size)           # :1221
self.weight = Parameter(torch.empty(
    self.output_size, self.input_size_per_partition,                     # [out, in/p]
    device=torch.cuda.current_device(), dtype=config.params_dtype))
_initialize_affine_weight_gpu(self.weight, init_method, partition_dim=1, ...)  # :1264
```

### Q2 🟢 列切前向不通信、行切前向要 all-reduce，本质差异是什么？

**答案**

关键看**输入是否复制、输出是否是部分和**：

- **列切**：输入 `X` 在各卡**完整复制**，各卡算出输出的不同列块，拼接即完整结果 → 前向不通信（`gather_output=False`）。
- **行切**：输入沿最后一维切分，每卡算出的 `Yᵢ` 只是**沿收缩维（contraction dim）的部分和**，必须 all-reduce 求和 → `layers.py:1357`。

一句话：**行切在矩阵乘法的收缩维上切分，部分和必须求和；列切在非收缩维切分，天然可拼接。**

**Megatron 实现**：行切前向末尾的 all-reduce（`layers.py` `RowParallelLinear.forward`）：

```python
output_parallel = self._forward_impl(input=input_parallel, weight=self.weight, ...)  # 各卡部分和
# ... SP / expert 分支略 ...
output_ = reduce_from_tensor_model_parallel_region(output_parallel, group=self.tp_group)  # :1357 g/all-reduce
```

列切前向则**没有**这一步——`ColumnParallelLinear.forward` 默认 `gather_output=False` 直接返回分片 `output_parallel`（`layers.py:1098-1099`）。

## 二、核心机制题：f / g 共轭算子

### Q3 🟡 解释 `f` 和 `g` 两个算子，它们前向 / 反向各做什么？为什么是「共轭 / 伴随」关系？

**答案**

定义在 `mappings.py`：

| 算子 | 类 | 前向 | 反向 | 位置 |
|---|---|---|---|---|
| **f** | `_CopyToModelParallelRegion` | identity | all-reduce | `mappings.py:197` |
| **g** | `_ReduceFromModelParallelRegion` | all-reduce | identity | `mappings.py:217` |

- **f 放在列切入口**：前向复制（identity），反向 all-reduce（复制的 `X` 喂给所有列分片，`dX = ΣdXᵢ` 跨卡求和）。
- **g 放在行切出口**：前向 all-reduce（部分和求和），反向 identity（梯度天然正确分片）。

**共轭关系**：「向 N 卡复制」的伴随（转置）算子就是「all-reduce 求和」，反之亦然。规律：**前向 copy ⇒ 反向 sum；前向 sum ⇒ 反向 copy**。把 f、g 放在两个边界，autograd 自动得到正确梯度，且**一对「列→行」前向仅 1 次 all-reduce（g）、反向仅 1 次 all-reduce（f）**。

```python
# f: _CopyToModelParallelRegion            # mappings.py:197
def forward(ctx, input_, group):  return input_
def backward(ctx, grad_output):   return _reduce(grad_output, ctx.group), None

# g: _ReduceFromModelParallelRegion         # mappings.py:217
def forward(ctx, input_, group):  return _reduce(input_, group)
def backward(ctx, grad_output):   return grad_output, None
```

### Q4 🟡 `mappings.py` 还有 `_ScatterToModelParallelRegion` / `_GatherFromModelParallelRegion`，它们和 f/g 什么关系？何时用？

**答案**

它们是**沿最后一维**做 split/gather 的另一对**共轭**算子：

| 类 | 前向 | 反向 |
|---|---|---|
| `_ScatterToModelParallelRegion` | split 最后一维 | gather 最后一维 |
| `_GatherFromModelParallelRegion` | all-gather 最后一维 | split 最后一维 |

使用场景：
- **`gather_output=True`**（列切想输出完整张量）：出口用 `_GatherFrom…` all-gather → `layers.py:1095`。
- **`input_is_parallel=False`**（行切拿到完整输入需自己切）：入口用 `_ScatterTo…` → `layers.py:1324`。

注意：标准 MLP/Attention 用 `gather_output=False` + `input_is_parallel=True`，这对算子**不触发**——正是为了省掉额外 gather/scatter。

**Megatron 实现**（`megatron/core/tensor_parallel/mappings.py`）：

```python
class _ScatterToModelParallelRegion(torch.autograd.Function):     # :236
    @staticmethod
    def forward(ctx, input_, group):
        ctx.group = group
        return _split_along_last_dim(input_, group)               # 前向 split 最后一维
    @staticmethod
    def backward(ctx, grad_output):
        return _gather_along_last_dim(grad_output, ctx.group), None  # 反向 all-gather

class _GatherFromModelParallelRegion(torch.autograd.Function):    # :256
    @staticmethod
    def forward(ctx, input_, group):
        ctx.group = group
        return _gather_along_last_dim(input_, group)              # 前向 all-gather
    @staticmethod
    def backward(ctx, grad_output):
        return _split_along_last_dim(grad_output, ctx.group), None   # 反向 split
```

## 三、组合应用题：MLP 与 Attention

### Q5 🟡 画出 MLP（h → 4h → h）的 TP 结构，说明各步切分与通信。为什么中间激活函数不需要通信？

**答案**

```
X[s,b,h] (复制)
   │  f  (前向 identity / 反向 all-reduce)
   ▼
linear_fc1 = ColumnParallelLinear(gather_output=False)   # h → 4h/p     mlp.py:221
   ▼  输出 [s,b,4h/p] (分片)
GeLU   ← 逐元素运算，在分片上直接算，无通信
   ▼  [s,b,4h/p] (分片)
linear_fc2 = RowParallelLinear(input_is_parallel=True)   # 4h/p → h     mlp.py:244
   │  g  (前向 all-reduce / 反向 identity)
   ▼
Y[s,b,h] (复制)
```

**激活函数为何不通信**：GeLU 是**逐元素**运算，`GeLU(Yᵢ)` 只依赖本卡的列块 `Yᵢ`。把它夹在列切与行切之间，中间段完全无通信——这正是「列切在前、行切在后」的精髓。

**通信次数**：前向 1 次 all-reduce（fc2 的 g），反向 1 次（fc1 的 f）。

**Megatron 实现**（`megatron/core/transformer/mlp.py` `MLP.__init__` / `forward`）：

```python
# fc1：列切，gather_output=False → 输出保持 [s,b,4h/p] 分片
self.linear_fc1 = submodules.linear_fc1(
    self.input_size, ffn_hidden_size, config=self.config,
    gather_output=False, skip_bias_add=True, tp_group=tp_group, ...)   # :216-229

# fc2：行切，input_is_parallel=True → 直接吃 fc1 的分片输出，出口 all-reduce
self.linear_fc2 = submodules.linear_fc2(
    self.config.ffn_hidden_size, self.config.hidden_size, config=self.config,
    input_is_parallel=True, skip_bias_add=True, tp_group=tp_group, ...)  # :236-250

def forward(self, hidden_states, ...):
    intermediate_parallel, bias_parallel = self.linear_fc1(hidden_states)  # [s,b,4h/p]
    intermediate_parallel = self.activation_func(intermediate_parallel)    # 逐元素，无通信
    output, output_bias = self.linear_fc2(intermediate_parallel)           # [s,b,h]，g/all-reduce
    return output, output_bias
```

### Q6 🟡 自注意力如何切？为什么 core attention（softmax(QKᵀ/√d)V）不需要跨卡通信？

**答案**

- **QKV 投影** `linear_qkv`：`ColumnParallelLinear(gather_output=False)`，按**注意力头**切分，每卡拿到完整的若干头 → `attention.py:1414`。
- **Core attention**：`softmax(QKᵀ/√d)·V` **按头独立**计算，每个头只用本卡的 Q/K/V，自成闭环，**零通信**。
- **输出投影** `linear_proj`：`RowParallelLinear(input_is_parallel=True)`，拼回并 all-reduce → `attention.py:400`。

结构与 MLP 同构：列切（按头）→ 免通信核心计算 → 行切（all-reduce）。前向 1 次 + 反向 1 次 all-reduce。

**Megatron 实现**（`megatron/core/transformer/attention.py`）：

```python
# QKV 投影：列切，gather_output=False，按注意力头切分            SelfAttention.__init__
self.linear_qkv = submodules.linear_qkv(
    self.config.hidden_size, self.linear_qkv_out_dim, config=self.config,
    gather_output=False, tp_group=tp_group, ...)                  # :1409-1420

# 输出投影：行切，input_is_parallel=True，出口 all-reduce         Attention.__init__
self.linear_proj = submodules.linear_proj(
    self.query_projection_size, self.config.hidden_size, config=self.config,
    input_is_parallel=True, tp_group=tp_group, ...)               # :394-405

def forward(self, hidden_states, ...):
    mixed_qkv, _ = self.linear_qkv(hidden_states)                 # 每卡拿到完整的若干头
    q, k, v = split(mixed_qkv)
    core_attn_out = self.core_attention(q, k, v, ...)             # 按头独立，零通信
    output, bias = self.linear_proj(core_attn_out)               # g/all-reduce
    return output, bias
```

### Q7 🟡 一个完整 Transformer 层（Attention + MLP），前向和反向各有几次 all-reduce？为什么说是理论最小值？

**答案**

- Attention：前向 1 + 反向 1
- MLP：前向 1 + 反向 1
- **合计：前向 2 次、反向 2 次 all-reduce。**

**为什么最小**：每个「列切→行切」子结构前向至少 1 次求和（收缩维被切的部分和）、反向至少 1 次求和（复制输入的梯度求和）。Attention 与 MLP 各是一个这样的子结构，无法再减。逐元素 / 按头运算夹在中间，避免了额外 gather/scatter。对应 [02.5 §7](../02.5-张量并行实现详解.md)。

**伪代码**（一个 Transformer 层的前向，标出通信点）：

```python
def transformer_layer_forward(x):          # x: [s,b,h] 复制态
    # --- Self-Attention 子结构 ---
    a = layernorm(x)                        # 逐元素，无通信
    a = self_attention(a)                   # 内含: 列切QKV(无通信) → core-attn(无通信)
                                            #       → 行切proj → all-reduce ①（前向第1次）
    x = x + a                               # 残差，无通信
    # --- MLP 子结构 ---
    m = layernorm(x)                        # 逐元素，无通信
    m = mlp(m)                              # 内含: 列切fc1(无通信) → GeLU(无通信)
                                            #       → 行切fc2 → all-reduce ②（前向第2次）
    x = x + m
    return x
# 前向 2 次 all-reduce（①②）；反向对称地在两个列切入口(f)各 1 次，共 2 次。
```

## 四、序列并行（SP）

### Q8 🟡 SP 解决什么问题？它把 TP 的 all-reduce 换成了什么？通信量变了吗？

**答案**

**动机**：标准 TP 中 LayerNorm / Dropout / 残差等**逐元素区域**的激活在每卡是**完整复制**的，占大量激活显存。SP 把这些区域激活也沿**序列维（第一维）**切分，降低峰值激活显存。

**核心恒等式**：`AllReduce = ReduceScatter + AllGather`（[02.3 §2.1](../02.3-通信原语速查.md)）。

- **行切出口**：`all-reduce` → `reduce_scatter_to_sequence_parallel_region`（沿序列维 reduce-scatter，结果分片）→ `layers.py:1353`。
- **列切入口 / GEMM 内部**：需完整序列时沿序列维 `all-gather`（在 GEMM 的 autograd 内完成）。
- 反向对称：前向 all-gather 的反向是 reduce-scatter，反之亦然（`_GatherFromSequenceParallelRegion` / `_ReduceScatterToSequenceParallelRegion`，`mappings.py:296/351`）。

**通信量**：RS + AG 的**总量与一次 AllReduce 相同**，但**激活显存峰值更低**——等通信量换省显存。
**额外代价**：SP 下各卡看到不同 token，复制参数（LayerNorm/qk_layernorm）梯度汇总需多一次跨 TP 的 SUM all-reduce（[02.5 §10.3](../02.5-张量并行实现详解.md)）。

**Megatron 实现**：行切出口按 `sequence_parallel` 分流（`layers.py` `RowParallelLinear.forward`）：

```python
if self.explicit_expert_comm:
    output_ = output_parallel                                              # MoE 专家：此处不通信
elif self.sequence_parallel:
    output_ = reduce_scatter_to_sequence_parallel_region(                  # :1353 SP：reduce-scatter
        output_parallel, group=self.tp_group)
else:
    output_ = reduce_from_tensor_model_parallel_region(                    # :1357 普通TP：all-reduce
        output_parallel, group=self.tp_group)
```

列切入口的 all-gather 沉到融合 GEMM 内部（`layers.py` `LinearWithGradAccumulationAndAsyncCommunication.forward`）：

```python
if sequence_parallel:                                                      # :499
    dim_size = list(input.size()); dim_size[0] *= tp_group.size()
    all_gather_buffer = get_global_memory_buffer().get_tensor(dim_size, input.dtype, "mpu")
    dist_all_gather_func(all_gather_buffer, input, group=tp_group)         # 沿序列维补齐
    total_input = all_gather_buffer
else:
    total_input = input
output = torch.matmul(total_input, weight.t())
```

### Q9 🔴 SP/TP 异步重叠为什么依赖 `CUDA_DEVICE_MAX_CONNECTIONS=1`？

**答案**

SP/TP 用了**异步通信与计算重叠**（async all-gather 的同时算 wgrad）。`CUDA_DEVICE_MAX_CONNECTIONS=1` 强制所有 kernel 走**单一 CUDA stream 队列**，保证先发起的通信 kernel 一定排在依赖它的计算 kernel 之前执行，避免乱序用到未完成的通信结果。若允许多连接，通信可能被调度器延后，破坏正确性 / 性能。见 `layers.py` 中相关注释（`547/566/579` 附近）。

**Megatron 实现**（`layers.py` GEMM 的 `backward`，异步通信 + 依赖注释）：

```python
if ctx.sequence_parallel:                                     # 反向先异步 all-gather 补齐 input
    handle = dist_all_gather_func(all_gather_buffer, input, group=tp_group, async_op=True)
    # Here we rely on CUDA_DEVICE_MAX_CONNECTIONS=1 to ensure that the
    # gather is scheduled before the input gradient computation      # :547-548
    total_input = all_gather_buffer
grad_input = grad_output.matmul(weight)                       # 与上面的 all-gather 重叠
...
if ctx.allreduce_dgrad:                                       # 普通TP：异步 all-reduce dgrad
    handle = torch.distributed.all_reduce(grad_input, group=tp_group, async_op=True)  # :565
if ctx.sequence_parallel:                                     # SP：异步 reduce-scatter dgrad
    handle = dist_reduce_scatter_func(sub_grad_input, grad_input, group=tp_group, async_op=True)  # :576
# 之后再算 wgrad，让通信藏在 wgrad 计算背后 —— 顺序正确性全靠单连接保证
```

## 五、词表并行

### Q10 🟡 `VocabParallelEmbedding` 怎么做？为什么「不属于本卡的 token 置零再 all-reduce」能得正确结果？

**答案**

- 词表沿 vocab 维切到各卡，每卡负责 `[vocab_start_index, vocab_end_index)` → `layers.py:232`。
- 前向（`layers.py:284`）：① 构造 `input_mask` 标记越界 token；② id 减 `vocab_start_index`；③ 查表；④ **把不属于本卡的行置零**（`layers.py:306`）；⑤ `reduce_from_tensor_model_parallel_region` all-reduce 求和（`layers.py:324`，SP 下用 reduce-scatter）。

**为什么正确**：每个 token 只被**唯一一张卡**拥有，其余卡置零，求和 `0+0+真值+0 = 真值`，恰好还原。这是「独热求和」技巧。

**Megatron 实现**（`layers.py` `VocabParallelEmbedding.forward`，`:284`）：

```python
if self.tp_group.size() > 1:
    input_mask = (input_ < self.vocab_start_index) | (input_ >= self.vocab_end_index)  # 越界标记
    masked_input = input_.clone() - self.vocab_start_index         # 映射到本卡局部下标
    masked_input[input_mask] = 0                                   # 越界先夹到 0，避免非法索引
else:
    masked_input = input_
output_parallel = F.embedding(masked_input, self.weight)          # 本卡词表查表
if self.tp_group.size() > 1:
    output_parallel[input_mask, :] = 0.0                          # 不属于本卡的行置零
# reduce_scatter_embeddings 为 SP 分支；否则普通 all-reduce：
output = reduce_from_tensor_model_parallel_region(output_parallel, group=self.tp_group)  # :324
```

### Q11 🟡 为什么要用 `vocab_parallel_cross_entropy`？做几次通信？

**答案**

**动机**：词表极大（10 万+），先 all-gather 完整 logits 再算 CE 会产生巨大 `[s,b,vocab]` 通信与显存。词表并行下 logits 本就沿 vocab 切分，直接在分片上算更省。

**做法**（`cross_entropy.py:119`）：① 局部 max → **all-reduce MAX**（数值稳定）；② 掩码取本卡预测 logit → **all-reduce SUM** 汇总 `predicted_logits`；③ **all-reduce SUM** 汇总 `sum_exp_logits`；④ `loss = log(sum_exp) − predicted`。

**通信次数**：共 **3 次 all-reduce（1×MAX + 2×SUM）**，每次只通信标量或 `[s,b]`，**从不通信完整 `[s,b,vocab]`**。

**Megatron 实现**（`cross_entropy.py` `_VocabParallelCrossEntropy.forward`，`:119`）：

```python
# ① 数值稳定：局部 max 后 all-reduce MAX
vocab_parallel_logits, logits_max = ...calculate_logits_max(vocab_parallel_logits)
torch.distributed.all_reduce(logits_max, op=ReduceOp.MAX, group=tp_group)           # :130

# 取本卡词表范围，掩码出预测 logit / 局部 sum_exp
vocab_start_index, vocab_end_index = get_vocab_range(partition_vocab_size, rank, world_size)
(target_mask, masked_target_1d, predicted_logits, sum_exp_logits, exp_logits) = \
    ...calculate_predicted_logits(vocab_parallel_logits, target, logits_max,
                                  vocab_start_index, vocab_end_index)

# ②③ 跨卡汇总：predicted 与 sum_exp 各 all-reduce SUM（都只有 [s,b] 大小）
torch.distributed.all_reduce(predicted_logits, op=ReduceOp.SUM, group=tp_group)      # :146
torch.distributed.all_reduce(sum_exp_logits,  op=ReduceOp.SUM, group=tp_group)       # :150

# loss = log(sum_exp) - predicted
exp_logits, loss = ...calculate_cross_entropy_loss(exp_logits, predicted_logits, sum_exp_logits)
ctx.save_for_backward(exp_logits, target_mask, masked_target_1d)   # 反向直接用，无需重算
return loss
```

## 六、通信原语（5 种通信模式）

> 所有并行策略的通信都由集合原语拼成，务必吃透。详见 [02.3 通信原语速查](../02.3-通信原语速查.md)。

### Q12 🟢 说清楚这 5 种通信模式：Broadcast、All-Reduce、All-Gather、Reduce-Scatter、All-to-All——各自「做完后每张卡手上有什么」？分别用在哪种并行？

**答案**

| 模式 | 语义（做完后每卡有什么） | 每卡通信量 | 主要用途 |
|------|--------------------------|:--:|----------|
| **Broadcast** | root 的同一份数据发给所有卡，人手一份且相同 | `(N−1)·S` | 加载 / 分发权重、同步同一 batch |
| **All-Reduce** | 各卡数据求和后，**所有卡拿到相同的完整和** | **≈ 2S**（几乎与 N 无关） | **DP 同步梯度**、**TP 合并部分和 / 激活** |
| **All-Gather** | 各持一片 → **所有卡都拿到完整拼接** | `(N−1)·S` | **SP 补齐序列**、ZeRO-3 聚合参数 |
| **Reduce-Scatter** | 先求和再切片，每卡只留**和的 1/N** | `(N−1)·S` | **SP**、分布式优化器切梯度 |
| **All-to-All** | 每卡把不同块发给每张卡（如矩阵转置） | ∝ 数据量 | **MoE 的 dispatch / combine** |

**必记恒等式**：`All-Reduce = Reduce-Scatter + All-Gather` → 解释了 AllReduce 通信量 ≈2S、几乎与卡数无关，也是 SP / ZeRO「通信量守恒换显存」的根基。

**原语 → 策略速查**：DP=AllReduce（或 RS+AG）｜TP=AllReduce｜SP=AllGather+ReduceScatter｜PP=Send/Recv（P2P）｜CP=Ring Send/Recv 或 All-to-All｜EP=All-to-All×2。

**Megatron 实现**：底层原语在 `mappings.py` 里都是对 `torch.distributed` 的薄封装：

```python
# All-Reduce（TP 合并部分和）                                  mappings.py:22
def _reduce(input_, group):
    if group.size() == 1: return input_
    torch.distributed.all_reduce(input_.contiguous(), group=group)
    return input_

# All-Gather 沿第 0 维（SP 补齐序列）                            mappings.py:114
def _gather_along_first_dim(input_, group, ...):
    dim_size = list(input_.size()); dim_size[0] *= group.size()
    output = torch.empty(dim_size, dtype=input_.dtype, device=torch.cuda.current_device())
    dist_all_gather_func(output, input_.contiguous(), group=group)   # all_gather_into_tensor
    return output

# Reduce-Scatter 沿第 0 维（SP / 分布式优化器）                   mappings.py:155
def _reduce_scatter_along_first_dim(input_, group, ...):
    dim_size = list(input_.size()); dim_size[0] //= group.size()
    output = torch.empty(dim_size, dtype=input_.dtype, device=torch.cuda.current_device())
    dist_reduce_scatter_func(output, input_.contiguous(), group=group)  # reduce_scatter_tensor
    return output
```

> Broadcast/All-to-All 分别落在权重初始化广播、MoE `token_dispatcher`；PP 的 Send/Recv 在 `pipeline_parallel/p2p_communication.py`。

### Q13 🔴 Reduce-Scatter 的「求和」到底在哪张卡上算？为什么每卡通信量只有 ≈S 而不是 (N−1)·S？

**答案**

**不是**把所有数据搬到某张卡上求和。NCCL 用 **Ring 算法**把求和**摊到全组每张卡**：数据沿环逐跳流动，**每一跳接收方就把收到的分片加到本地对应分片上**（reduce-on-receive，边传边加）。

以 `N=4`、追踪最终归 card0 的「a 片」为例：

```
环：card0 → card1 → card2 → card3 →(回) card0
第1跳 card1──a1──►card2 ： card2 算 a1+a2
第2跳 card2─(a1+a2)─►card3： card3 算 a1+a2+a3
第3跳 card3─(…+a3)─►card0： card0 算 …+a0 = 全和
```

b/c/d 片同时错位并行滚动，故每卡每步只收 / 加 / 发一个分片。因为「边传边加、每卡只经手 (N−1)/N 份数据」，每卡通信量才 **≈S**，且计算无中心瓶颈——这是它能扩到成百上千卡的根本。

**伪代码**（Ring Reduce-Scatter，`reduce-on-receive` 边传边加）：

```python
# N 张卡，每卡持完整数组切成 N 片：chunk[0..N-1]；最终卡 r 只保留第 r 片的全和
def ring_reduce_scatter(local_chunks, rank, N, send, recv):
    acc = local_chunks                      # 本地 N 片
    for step in range(N - 1):               # 共 N-1 跳
        send_idx = (rank - step) % N        # 本步要发出的片
        recv_idx = (rank - step - 1) % N    # 本步要收进的片
        send(acc[send_idx], to=(rank + 1) % N)
        incoming = recv(frm=(rank - 1) % N)
        acc[recv_idx] += incoming           # ★ 加法发生在“接收卡”上（reduce-on-receive）
    return acc[(rank + 1) % N]              # 卡 r 最终持有的那一片的全和
# 每步只发/收 1 片 → 每卡通信量 (N-1)·S/N ≈ S；求和均摊到全组，无中心瓶颈。
```

真实实现由 NCCL 完成，Megatron 侧只调用 `torch.distributed.reduce_scatter_tensor`（见 Q12 的 `_reduce_scatter_along_first_dim`）。

## 七、其他并行方式

### Q14 🟡 简述 PP 的 1F1B 调度，它和 TP 的通信有何本质区别？

**答案**

- **PP**：按**层**切成多个 stage，stage 间用 **P2P send/recv** 传激活 / 梯度（`pipeline_parallel/schedules.py`、`p2p_communication.py`）。
- **1F1B（one-forward-one-backward）**：分 warmup / steady / cooldown 三段，稳态期每完成一个 micro-batch 前向就紧接一个反向，使**气泡（bubble）最小**、激活显存受控；交错（interleaved）版进一步拆 virtual stage 缩小气泡。

**与 TP 区别**：TP 用**集合通信（all-reduce）**、在**层内**、频繁且在关键路径，要求高带宽（NVLink 内）；PP 用**点对点（P2P）**、在**层间**、通信量小频率低，适合跨节点。二者常组合（TP 节点内、PP 跨节点）。

**Megatron 实现**（`pipeline_parallel/schedules.py` `forward_backward_pipelining_without_interleaving`，简化）：

```python
# warmup 数 = 本 stage 到最后一个 stage 的距离；越靠前的 stage 需先攒越多前向
num_warmup_microbatches = pipeline_parallel_size - pipeline_parallel_rank - 1     # :887
num_warmup_microbatches = min(num_warmup_microbatches, total_num_microbatches)
num_microbatches_remaining = total_num_microbatches - num_warmup_microbatches

# ① Warmup：只前向，填满流水线
for i in range(num_warmup_microbatches):
    inp = recv_forward(...)
    out = forward_step(data_iterator, model, inp, ...)
    send_forward(out, ...)

# ② Steady：1F1B —— 每个前向后紧跟一个反向，激活即用即释放
for i in range(num_microbatches_remaining):
    out = forward_step(data_iterator, model, recv_forward(...), ...)
    grad = send_forward_recv_backward(out, ...)        # 前向送出 + 收回对应反向梯度
    inp_grad = backward_step(..., grad, ...)
    send_backward(inp_grad, ...)

# ③ Cooldown：只反向，排空流水线
for i in range(num_warmup_microbatches):
    backward_step(..., recv_backward(...), ...)
    send_backward(...)
```

`send_forward` / `recv_backward` 等最终落到 `p2p_communication.py` 的 `torch.distributed.isend/irecv`。

### Q15 🟡 EP（MoE）的通信模式是什么？用什么原语，和 TP 有何不同？

**答案**

- **EP**：把多个 expert 分到不同卡（`get_expert_model_parallel_group`）。router 为每个 token 选 top-k expert，用 **all-to-all** 把 token 分发（dispatch）到持有对应 expert 的卡，算完再 all-to-all 收回（combine）→ `transformer/moe/`（`router.py`、`token_dispatcher.py`、`fused_a2a.py`）。
- **与 TP 不同**：TP 切**单个权重矩阵**、用 all-reduce、静态确定；EP 切**整组 expert**、用 all-to-all、通信量**数据依赖**（路由结果决定，可能负载不均）。

**Megatron 实现**（`transformer/moe/token_dispatcher.py`，`MoEAllToAllTokenDispatcher`）：

```python
from .moe_utils import all_to_all

# dispatch：按 router 结果，把本地 token 送到“持有其目标专家”的卡      :678
global_input_tokens = all_to_all(
    self.ep_group, permutated_local_input_tokens,          # 已按目标专家排好序
    self.output_splits, self.input_splits)                 # 变长 split（负载不均→数据依赖）
global_probs = all_to_all(self.ep_group, permutated_probs, ...)   # :691

# ... 本卡专家计算 expert(global_input_tokens) ...

# combine：算完再 all-to-all 送回原卡（dispatch 的逆）             :837
permutated_local_input_tokens = all_to_all(
    self.ep_group, hidden_states, self.input_splits, self.output_splits)
```

一次 MoE 层 = **2 次 all-to-all**（dispatch + combine）。`all_to_all` 底层是 `torch.distributed.all_to_all_single`。

### Q16 🟡 CP 解决什么问题？和 SP 有什么区别？

**答案**

- **CP（上下文并行）**：面向**超长上下文**，把序列维切到各卡，但在 **attention 内部**用 ring / all-gather 通信，让每卡都能 attend 到**完整序列**（attention 需要全局 K/V）→ `transformer/dot_product_attention.py`、`get_context_parallel_group()`。
- **和 SP 区别**：SP 只在 TP 内部、对逐元素区域沿序列切分省激活，不改变单卡序列长度上限；**CP 是独立并行维度**，专为把超长序列摊到多卡、突破单卡序列长度而设计，需在 attention 里做跨卡 K/V 通信（Ring Attention）。

**伪代码**（Ring Attention：每卡持 1/N 序列的 Q/K/V，环形传 K/V 累积 online-softmax）：

```python
# CP=N，每卡 rank 持有序列的第 rank 段：q_local, k_local, v_local
def ring_attention(q_local, k_local, v_local, cp_group, N, rank):
    k, v = k_local, v_local
    out, m, l = init_online_softmax(q_local)      # FlashAttention 式在线累积 (max, sumexp)
    for step in range(N):                          # 环上转一圈，见过所有 K/V 段
        out, m, l = flash_attn_update(q_local, k, v, out, m, l, causal_mask(step))
        if step < N - 1:                           # 把当前 K/V 传给下一个邻居、收上游的
            k = ring_send_recv(k, send_to=(rank+1)%N, recv_from=(rank-1)%N, group=cp_group)
            v = ring_send_recv(v, ...)
    return finalize(out, l)                         # 每卡得到本地 Q 段对全序列的注意力
```

**Megatron 实现**：CP 逻辑在 `transformer/dot_product_attention.py` + TE 的 attention 后端，进程组来自 `parallel_state.get_context_parallel_group()`；K/V 环形交换用 P2P（`batch_isend_irecv`），Zigzag 分片做因果掩码下的负载均衡（见 [02.7](../02.7-上下文并行.md)）。

### Q17 🟡 阅读下列源码，指出 `f`、`g` 各自前向 / 反向的通信，并说明为何一个前向不通信、一个前向要通信。

```python
# megatron/core/tensor_parallel/mappings.py
def _reduce(input_, group):
    if group.size() == 1:
        return input_
    torch.distributed.all_reduce(input_.contiguous(), group=group)
    return input_

class _CopyToModelParallelRegion(torch.autograd.Function):        # ← A
    @staticmethod
    def forward(ctx, input_, group):
        ctx.group = group
        return input_
    @staticmethod
    def backward(ctx, grad_output):
        return _reduce(grad_output, ctx.group), None

class _ReduceFromModelParallelRegion(torch.autograd.Function):     # ← B
    @staticmethod
    def forward(ctx, input_, group):
        return _reduce(input_, group)
    @staticmethod
    def backward(ctx, grad_output):
        return grad_output, None
```

**答案**

- **A = `f`**：前向 `return input_` → **identity，不通信**；反向 `_reduce(grad_output)` → **all-reduce**。用于**列切入口**：前向输入是复制的无需动，反向多个列分片对同一份 `X` 的梯度要求和 `dX=ΣdXᵢ`。
- **B = `g`**：前向 `_reduce(input_)` → **all-reduce**；反向 `return grad_output` → **identity，不通信**。用于**行切出口**：前向把各卡沿收缩维的部分和求和，反向梯度对复制输出天然正确分片、无需通信。

二者互为伴随：**前向 copy ⇒ 反向 sum；前向 sum ⇒ 反向 copy**。

### Q18 🟡 阅读 `RowParallelLinear.forward` 片段，回答：`input_is_parallel` 为 True / False 时行为差异？普通 TP 与 SP 下出口通信分别是什么？

```python
# megatron/core/tensor_parallel/layers.py  (RowParallelLinear.forward)
if self.input_is_parallel:
    input_parallel = input_
else:
    assert not self.sequence_parallel
    input_parallel = scatter_to_tensor_model_parallel_region(input_, group=self.tp_group)

output_parallel = self._forward_impl(input=input_parallel, weight=self.weight, bias=None, ...)

elif self.sequence_parallel:
```

  - 普通 TP → `reduce_from_tensor_model_parallel_region`（**all-reduce**，即 g）。

### Q19 🔴 阅读 `ColumnParallelLinear.forward` 片段，回答：为什么这里对 `input_` 有时套 `copy_to_tensor_model_parallel_region`（f）、有时不套？`gather_output` 起什么作用？

```python
# megatron/core/tensor_parallel/layers.py  (ColumnParallelLinear.forward)
if (self.allreduce_dgrad or self.sequence_parallel
        or self.explicit_expert_comm or self.disable_grad_reduce):
    input_parallel = input_
else:
    input_parallel = copy_to_tensor_model_parallel_region(input_, group=self.tp_group)

output_parallel = self._forward_impl(input=input_parallel, weight=weight, bias=bias,
                                     allreduce_dgrad=allreduce_dgrad,
                                     sequence_parallel=..., tp_group=self.tp_group)

gather_output = self.gather_output
if runtime_gather_output is not None:
    gather_output = runtime_gather_output
if gather_output:
    output = gather_from_tensor_model_parallel_region(output_parallel, group=self.tp_group)
else:
    output = output_parallel
```

**答案**

- **何时套 f**：默认路径（非 SP、非 fused-dgrad、非专家）套 `copy_to_tensor_model_parallel_region`（f），保证反向对复制输入的梯度 all-reduce。
- **何时不套**：当 `sequence_parallel` / `allreduce_dgrad` / `explicit_expert_comm` 等为真时，`input_parallel = input_`。因为此时反向的 dgrad 通信被**下沉到融合 GEMM 内部**（由 `allreduce_dgrad` 控制），或 SP 路径改在 GEMM 内做 all-gather / reduce-scatter，无需再在此处套 f，避免重复通信。
- **`gather_output`**：
  - `False`（默认）→ 输出保持分片 `[s,b,output_size/p]`，直接喂给下游 RowParallelLinear（省一次通信）。
  - `True` → `gather_from_tensor_model_parallel_region` all-gather 还原完整 `[s,b,output_size]`（用于层边界或需要完整输出的场景）。

### Q20 🔴 下面是词表并行 embedding 的核心片段，解释「mask → 置零 → all-reduce」为什么能得到正确的完整 embedding。

```python
# megatron/core/tensor_parallel/layers.py  (VocabParallelEmbedding.forward)
input_mask = (input_ < self.vocab_start_index) | (input_ >= self.vocab_end_index)
masked_input = input_.clone() - self.vocab_start_index
masked_input[input_mask] = 0
output_parallel = self.weight[masked_input]      # 本卡词表查表
output_parallel[input_mask, :] = 0.0             # 不属于本卡的 token 行置零
output = reduce_from_tensor_model_parallel_region(output_parallel, group=self.tp_group)
```

**答案**

词表沿 vocab 维切分，每个 token id 只落在**唯一一张卡**的 `[vocab_start_index, vocab_end_index)` 内：
1. `input_mask` 标出**不属于本卡**的 token；
2. 减去 `vocab_start_index` 把 id 映射到本卡局部下标，越界的先夹到 0 避免非法索引；
3. 查表后，把 `input_mask` 命中的行**置零**——这些 token 不归本卡，本卡对它们「没有贡献」；
4. 跨 TP `all-reduce` 求和：对每个 token，只有拥有它的那张卡是真值、其余都是 0，求和得 `0+…+真值+…+0 = 真值`。

本质是**独热求和**：每 token 恰好一处非零，SUM 后精确还原。SP 下把最后的 all-reduce 换成 reduce-scatter（结果沿序列分片）。

### Q21 🟡 手写 `f` 与 `g` 两个 `autograd.Function`（列切入口 / 行切出口），使前向反向通信都正确。

**要求**：`f` 前向 identity、反向 all-reduce；`g` 前向 all-reduce、反向 identity。给出可运行的最小实现。

**参考答案**

```python
import torch
import torch.distributed as dist

def _all_reduce(x, group):
    # 单卡直通，避免无谓通信
    if group is None or group.size() == 1:
        return x
    x = x.contiguous()
    dist.all_reduce(x, op=dist.ReduceOp.SUM, group=group)
    return x

class _CopyToTP(torch.autograd.Function):
    """f：列切入口。前向复制（identity），反向 all-reduce。"""
    @staticmethod
    def forward(ctx, x, group):
        ctx.group = group
        return x                                   # identity
    @staticmethod
    def backward(ctx, grad):
        return _all_reduce(grad, ctx.group), None  # dX = Σ dXi

class _ReduceFromTP(torch.autograd.Function):
    """g：行切出口。前向 all-reduce，反向 identity。"""
    @staticmethod
    def forward(ctx, x, group):
        return _all_reduce(x, group)               # 合并部分和
    @staticmethod
    def backward(ctx, grad):
        return grad, None                          # 梯度天然正确分片

def copy_to_tp(x, group):    return _CopyToTP.apply(x, group)
def reduce_from_tp(x, group): return _ReduceFromTP.apply(x, group)
```

**要点**：`backward` 必须为每个 `forward` 的输入返回一个梯度（`group` 非张量 → 返回 `None`）；单卡短路；`all_reduce` 原地操作前先 `contiguous()`。

### Q22 🟡 基于 `f`/`g`，实现一个 TP 版 MLP（h → 4h → h），要求前向仅 1 次 all-reduce、反向仅 1 次 all-reduce。

**参考答案**

```python
import torch
import torch.nn as nn
import torch.nn.functional as F

class ColumnParallelLinear(nn.Module):
    """列切：权重按输出维切，输出保持分片（gather_output=False）。"""
    def __init__(self, in_features, out_features, group):
        super().__init__()
        self.group = group
        world = group.size() if group is not None else 1
        assert out_features % world == 0
        self.out_per_partition = out_features // world
        # 存储 [out/p, in]，对应 Y = X A^T 的列切
        self.weight = nn.Parameter(torch.empty(self.out_per_partition, in_features))
        nn.init.kaiming_uniform_(self.weight)

    def forward(self, x):                       # x: [s, b, in] 复制
        x = copy_to_tp(x, self.group)           # f：反向 all-reduce dX
        return F.linear(x, self.weight)         # [s, b, out/p] 分片

class RowParallelLinear(nn.Module):
    """行切：权重按输入维切，输入已分片（input_is_parallel=True）。"""
    def __init__(self, in_features, out_features, group):
        super().__init__()
        self.group = group
        world = group.size() if group is not None else 1
        assert in_features % world == 0
        self.in_per_partition = in_features // world
        self.weight = nn.Parameter(torch.empty(out_features, self.in_per_partition))
        nn.init.kaiming_uniform_(self.weight)
        self.bias = nn.Parameter(torch.zeros(out_features))

    def forward(self, x):                       # x: [s, b, in/p] 分片
        y = F.linear(x, self.weight)            # 各卡部分和 [s, b, out]
        y = reduce_from_tp(y, self.group)       # g：前向 all-reduce
        return y + self.bias                    # bias 不切，reduce 后再加

class TPMLP(nn.Module):
    def __init__(self, hidden, group, ffn_mult=4):
        super().__init__()
        inner = hidden * ffn_mult
        self.fc1 = ColumnParallelLinear(hidden, inner, group)  # h -> 4h/p
        self.fc2 = RowParallelLinear(inner, hidden, group)     # 4h/p -> h

    def forward(self, x):                        # [s, b, h] 复制
        x = self.fc1(x)                          # [s, b, 4h/p] 分片
        x = F.gelu(x)                            # 逐元素，分片上直接算，无通信
        x = self.fc2(x)                          # [s, b, h]，出口 1 次 all-reduce
        return x
```

**通信核对**：
- 前向：`fc1` 的 f 是 identity（不通信）；GeLU 逐元素（不通信）；`fc2` 的 g 是 **1 次 all-reduce**。✅
- 反向：`fc2` 的 g 反向 identity（不通信）；`fc1` 的 f 反向 **1 次 all-reduce**（`dX=ΣdXᵢ`）。✅
- 关键点：`bias` 在 reduce 之后再加（否则会被 all-reduce 放大 world 倍）；`fc1` 输出**不 gather**，直接作为 `fc2` 的分片输入。

### Q23 🔴 用 `reduce_scatter` + `all_gather` 手写「序列并行版」的行切出口与列切入口，验证 `AllReduce = ReduceScatter + AllGather`。

**参考答案**

```python
import torch
import torch.distributed as dist

def _reduce_scatter_first_dim(x, group):
    """沿第 0 维（序列维）reduce-scatter：先求和，每卡留 1/N。"""
    world = group.size()
    if world == 1:
        return x
    assert x.size(0) % world == 0
    out_shape = list(x.size()); out_shape[0] //= world
    out = torch.empty(out_shape, dtype=x.dtype, device=x.device)
    dist.reduce_scatter_tensor(out, x.contiguous(), op=dist.ReduceOp.SUM, group=group)
    return out                                   # [s/N, b, h] 分片且已求和

def _all_gather_first_dim(x, group):
    """沿第 0 维 all-gather：各卡分片拼成完整序列。"""
    world = group.size()
    if world == 1:
        return x
    out_shape = list(x.size()); out_shape[0] *= world
    out = torch.empty(out_shape, dtype=x.dtype, device=x.device)
    dist.all_gather_into_tensor(out, x.contiguous(), group=group)
    return out                                   # [s, b, h] 完整

class _ReduceScatterToSP(torch.autograd.Function):
    """行切出口（SP）：前向 reduce-scatter，反向 all-gather。"""
    @staticmethod
    def forward(ctx, x, group):
        ctx.group = group
        return _reduce_scatter_first_dim(x, group)
    @staticmethod
    def backward(ctx, grad):
        return _all_gather_first_dim(grad, ctx.group), None

class _GatherFromSP(torch.autograd.Function):
    """列切入口（SP）：前向 all-gather，反向 reduce-scatter。"""
    @staticmethod
    def forward(ctx, x, group):
        ctx.group = group
        return _all_gather_first_dim(x, group)
    @staticmethod
    def backward(ctx, grad):
        return _reduce_scatter_first_dim(grad, ctx.group), None

# 等价性验证：对分片求和后拼回 == 直接 all-reduce
def check_equivalence(x_full, group):
    # 标准 TP：直接 all-reduce
    ar = x_full.clone().contiguous()
    dist.all_reduce(ar, op=dist.ReduceOp.SUM, group=group)
    # SP 分解：reduce-scatter 再 all-gather
    sp = _all_gather_first_dim(_reduce_scatter_first_dim(x_full, group), group)
    assert torch.allclose(ar, sp), "AllReduce != ReduceScatter + AllGather"
```

**要点**：
- 前向 `reduce-scatter` 的反向必是 `all-gather`，反之亦然——这对操作**互为伴随**（对照 `mappings.py:296/351`）。
- `AllReduce = ReduceScatter + AllGather`：`check_equivalence` 中 `sp` 与 `ar` 逐元素相等，通信总量与一次 all-reduce 相同，但前向后激活按序列维分片存储，省激活显存。
- 真实实现里 all-gather 常与后续 GEMM 的 wgrad 计算**异步重叠**，依赖 `CUDA_DEVICE_MAX_CONNECTIONS=1` 保证顺序（见 Q9）。

## 十、进阶追问

### Q24 🔴 如果把 MLP 写成「行切 → 列切」（顺序反过来）会怎样？

**答案**

会**破坏免通信中间段并增加通信**。行切在前意味着入口就要 all-reduce/scatter 得到完整结果，而中间 GeLU 非线性 `GeLU(ΣYᵢ) ≠ ΣGeLU(Yᵢ)`，必须先 all-reduce 还原完整激活再做 GeLU，再为下一个列切复制——凭空多出通信与显存。

正确顺序「列切 →（逐元素）→ 行切」保证中间激活始终是**可独立计算的列块**，逐元素运算在分片上与全量等价，通信压到每子结构 1 次。这解释了切分顺序不可随意。

**伪代码对比**：

```python
# ✅ 正确：列切 → 逐元素 → 行切（Megatron 的做法）
h = column_linear(x)          # [s,b,4h/p] 分片，无通信
h = gelu(h)                   # 逐元素，分片上 == 全量，无通信
y = row_linear(h)             # 出口 1 次 all-reduce
# 前向仅 1 次 all-reduce

# ❌ 反序：行切 → 逐元素 → 列切
h = row_linear(x)             # 部分和 → 必须 all-reduce 还原完整 h   （通信①）
h = gelu(h)                   # 非线性: GeLU(Σhᵢ) ≠ ΣGeLU(hᵢ)，只能在完整 h 上算
h = copy_to_tp(h)            # 为列切复制到各卡（反向还要 all-reduce）
y = column_linear(h)          # [s,b,out/p] 分片
# 平白多出 all-reduce，且激活变回完整态、显存更大
```

### Q25 🔴 反向传播时，列切层和行切层各在哪一步产生 all-reduce？为什么行切反向不通信？

**答案**

- **列切层**：入口 **f 反向 all-reduce**（复制输入 → 梯度跨卡求和 `dX=ΣdXᵢ`）→ `mappings.py:214`。
- **行切层**：出口 **g 反向 identity，不通信** → `mappings.py:233`。因为行切输出是复制的，梯度 `dY` 各卡相同，回传到各自权重分片 `Aᵢ`、输入分片 `Xᵢ` 时天然正确分片，无需求和。

对称性：**前向通信在行切出口（g），反向通信在列切入口（f）**，各一次，正是 f/g 伴随关系的体现。参见 [02.5 §10.2](../02.5-张量并行实现详解.md)。

**Megatron 实现**（`mappings.py`，反向通信落点一目了然）：

```python
class _CopyToModelParallelRegion(torch.autograd.Function):     # f：列切入口
    @staticmethod
    def forward(ctx, input_, group):  return input_            # 前向 identity
    @staticmethod
    def backward(ctx, grad_output):
        return _reduce(grad_output, ctx.group), None          # :214 反向 all-reduce ← 列切的通信

class _ReduceFromModelParallelRegion(torch.autograd.Function): # g：行切出口
    @staticmethod
    def forward(ctx, input_, group):  return _reduce(input_, group)  # 前向 all-reduce
    @staticmethod
    def backward(ctx, grad_output):
        return grad_output, None                              # :233 反向 identity ← 行切不通信
```

### Q26 🔴 TP、SP、PP、DP、EP、CP 六种并行，按「通信原语 / 切什么 / 通信位置 / 带宽要求」列一张对照总表。

**答案**

| 并行 | 切什么 | 主要原语 | 通信位置 | 带宽要求 / 部署 |
|------|--------|----------|----------|-----------------|
| **DP** | 切数据、复制模型 | All-Reduce（或 RS+AG） | 反向末尾同步梯度 | 中，可跨节点 |
| **TP** | 切单个权重矩阵 | All-Reduce | 层内、每层 2+2 次 | 高，NVLink 内 |
| **SP** | 在 TP 内切激活的序列维 | AllGather + ReduceScatter | 层内替换 TP 的 AllReduce | 高，同 TP |
| **PP** | 切层（stage） | Send/Recv（P2P） | 相邻 stage 之间 | 低，适合跨节点 |
| **EP** | 切 expert 组 | All-to-All ×2 | MoE dispatch/combine | 中高，数据依赖 |
| **CP** | 切序列（长上下文） | Ring Send/Recv 或 All-to-All | attention 内交换 K/V | 高 |

**组合经验**：节点内用 TP(+SP)、跨节点用 PP，再叠 DP 扩规模；MoE 叠 EP；超长序列叠 CP。每个原语都要指定 process group，划分见 [02.4](../02.4-并行组构建与通信详解.md)。

**Megatron 实现**：六个维度的进程组由 `parallel_state` 统一构建，各原语按 `group=` 落到对应维度：

```python
# megatron/core/parallel_state.py（构建后按维度取用）
tp_group = get_tensor_model_parallel_group()       # TP：all-reduce 的通信域
dp_group = get_data_parallel_group()               # DP：梯度 all-reduce
pp_group = get_pipeline_model_parallel_group()     # PP：相邻 stage send/recv
cp_group = get_context_parallel_group()            # CP：attention 内 K/V 环形交换
ep_group = get_expert_model_parallel_group()       # EP：token all-to-all

# 各并行调用同一批原语，只是 group 不同：
reduce_from_tensor_model_parallel_region(x, group=tp_group)   # TP
torch.distributed.all_reduce(grad, group=dp_group)            # DP
all_to_all(ep_group, tokens, ...)                             # EP
```

> rank 如何被划进各组（如 `tp=2, pp=2, dp=2` 共 8 卡的编号规则）是 [02.4](../02.4-并行组构建与通信详解.md) 的主题。
