# 超出框架的工程考量（容灾 / 可观测性 / 数值 / 物理 / 运维 / 安全）

> 本篇是《训练可靠性工程》文档集的**第三支**，承载 [01 总览](./01-可靠性工程总览与Goodput目标.md) 三层框架里 **容灾（DR）** 与 **支撑层** 中**超出 Megatron/NVRx 源码**的维度。
> - **完整需求心智图见 [01 §0](./01-可靠性工程总览与Goodput目标.md)**——本篇展开其中 **DR 支 + 超框架支撑**（标"进阶"的格子），以及跨界特性 [§C6 Pod/节点级重调度](#c6-作业编排与-pod--节点级重调度-超出代码) 的编排层。
> - 与 [02 高可用与容错](./02-高可用与容错.md) 的关系：02 严格以源码为锚（"Megatron 接线、NVRx 造机制"）；本篇刻意越过这条边界，覆盖到集群 / 基础设施 / 运维层。凡超出 Megatron 代码范围的条目标注 `[超出代码]`。
> - 概念定位与 RTO/RPO/三层框架见 [01](./01-可靠性工程总览与Goodput目标.md)。

每条按统一体例：**问题是什么 / 为什么万卡才凸显 / [02] 的缺口 / 手段 / 参考锚点**。

# Part B · [02] 已触及、但只做了一半的维度

## B1. 相关性 / 成批故障（correlated / blast-radius failures）→ 容灾

- **问题**：[02](./02-高可用与容错.md) 隐含"故障相互独立"。现实中一个**机架 / 电源 PDU / ToR 交换机 / 冷却单元**会一次带走多卡甚至整机架。
- **为什么万卡才凸显**：规模越大，共享同一物理故障域的 rank 越多，"一损俱损"的概率越高；这类相关性故障正是把 HA 推向 DR 的分界。
- **两个直接后果**：
  1. **打破 Young/Daly 的无记忆假设**（见 [01 §4](./01-可靠性工程总览与Goodput目标.md)）——故障不再独立均匀，最优存档间隔公式只能作参考。
  2. **[02 §3.2 副本 checkpoint 的放置必须"故障域感知"**：副本不能和主本落在同一机架 / 供电域，否则"跨节点副本"照样一起死。[02](./02-高可用与容错.md) 只讲了 `replication_jump=J` 的 **rank 间隔**，**没讲这个 J 要对齐物理故障域**。
- **手段**：blast radius 度量、故障域（fault domain）划分与隔离、拓扑感知的副本布局（副本落到不同机架 / 不同供电域）。
- **归属**：单卡抖动端 ≈ HA；整机架 / 供电域端 = **DR**。

## B2. 可观测性体系（[02] 横切 A 只是"检测原语"，缺"遥测 + 归因体系"）

- **问题**：[02 横切 A](./02-高可用与容错.md) 有心跳 / 段超时 / sniff test，但那是**点状探针**。万卡需要成体系的可观测性。
- **为什么万卡才凸显**：日志与指标量随 rank 数暴增；**跨 rank 归因**（"到底是哪张卡 / 哪条链路"）在万卡上从"看一眼"变成"大海捞针"。
- **[02] 的缺口**：把检测当"够用的原语"，没讲检测到之后**如何快速定位根因**的体系。补：
  - **三支柱**：metrics（DCGM：GPU 温度 / 功耗 / ECC / 降频）、logs（万卡日志聚合与关联）、traces（NCCL flight recorder / 通信依赖追踪，如 Mycroft）。
  - **loss / grad-norm 曲线实时告警**：训练轨迹异常是最早的软信号。
  - **隐性故障（implicit failures）是核心难点**：hang、性能 jitter、SDC——传统 timeout 抓不到，且 root-cause 难。ByteDance / Meta 都自建工具把归因从"天"降到"小时"。
- **归属**：**支撑层**（HA/DR 两支都靠它）。
- **参考**：[Robust LLM Training Infrastructure at ByteDance (arXiv 2509.16293)](https://arxiv.org/html/2509.16293v4)、[Mycroft (arXiv 2509.03018)](https://arxiv.org/pdf/2509.03018)。

## B3. 数值稳定性与可复现性（与 [02 §4] 正交：§4 是硬件算错，这里是算法 / 数值本身）

- **问题**：并非所有"算错"都是硬件 SDC，还有**算法 / 数值层**的失稳。
- **[02] 的缺口**：
  - **loss spike 的处置策略**：[02 §4](./02-高可用与容错.md) 只**检测** spiky loss，没讲**处置**——skip 当前 batch / rollback 到上个 ckpt / 临时降 LR / 跳过坏数据。这是训练团队每天在做的决策。
  - **混合精度的数值边界**：fp16/bf16/fp8 的上溢下溢、loss scaling、grad clipping、优化器状态精度。fp8 训练在万卡上的稳定性尤其新。
  - **bitwise 确定性 / 可复现**：跨重启、跨 world-size 变化时，data shuffle 顺序、RNG seed、并行归约顺序是否一致——**这直接决定 SDC 能不能归因**（不确定性会把"硬件算错"和"数值本就该抖"混在一起，正是 [02 §4](./02-高可用与容错.md) 换卡重算要区分的三态）。[02 §3.4](./02-高可用与容错.md) 讲了数据位置续接，但没上升到"确定性可复现"这个总要求。
- **归属**：**支撑层**。
- **参考**：[Understanding Silent Data Corruption in LLM Training (arXiv 2502.12340)](https://arxiv.org/pdf/2502.12340)。

# Part C · [02] 完全未覆盖的维度

## C1. 数据 / 存储层的 HA 与吞吐 → 容灾

- **问题**：[02 第三部分](./02-高可用与容错.md) 只覆盖"训练**状态**"（权重 / 优化器 / 迭代号），没覆盖训练**数据集本身**与**存储带宽**。
- **为什么万卡才凸显**：万卡**同时读**数据 / 万卡**并发写** TB 级 checkpoint 会形成 **IO 风暴**——存储常常是恢复的**真实瓶颈**（本地副本 ckpt 正是为绕开它，但 [02](./02-高可用与容错.md) 没把"存储是瓶颈"这个前提讲明）。
- **手段 / 要点**：
  - 训练数据集的可靠性（分布式 FS / 对象存储高可用）、**数据版本管理与可复现**。
  - checkpoint 的**异地 / 离线冷备**（经典 DR，RPO），分层存储、保留与清理策略、成本。
  - **data loader 崩溃 / hang**：一类独立于计算的故障源。
- **归属**：异地冷备 = **DR**；存储吞吐 = **支撑**。

## C2. 规模化本身引入的系统开销（scale-induced）

- **问题**：规模本身带来"没坏也慢 / 没坏也脆"的开销。
- **要点**：
  - **初始化 / rendezvous 时间**：万卡 NCCL 建连近 **O(N²)**、集结慢——Meta 专门优化 job startup time 来提 goodput。[02 §5.2](./02-高可用与容错.md) 讲了 rendezvous 机制，但没讲"启动时间本身是 goodput 的一大项"。
  - **拓扑感知的并行布局**：把通信密集的 TP 组放进同一 NVLink 域、DP/PP 跨机架，减少跨 spine 流量。既是性能也是可靠性（少跨越 = 少故障面）。
  - **配置 / 元数据管理**：**配置错误是最常见的故障源之一**（并行度、rank 映射、超时参数写错）。万卡上配置的一致性分发与校验本身是可靠性问题。
- **归属**：**支撑层**。

## C3. 物理基础设施：电 / 热 / 机房 `[超出代码]` → 容灾（GW 级独有的新问题）

- **功率振荡威胁电网**：同步训练的 compute/comm 交替造成**数十~数百 MW 功率摆动**，其 **0.2–3 Hz** 频谱可能与汽轮机轴系 / 长输电线谐振，引发次同步谐振（SSR）、电压闪变。2025 年起成热点。
  - 缓解：**GPU 功率平滑**（Oracle AET 毫秒级心跳）、**机架级储能**（BESS + 超级电容）、保守封顶 GPU 功率。
- **热降频（thermal throttling）**：散热不足会把一张好卡变成 straggler——与 [02 §1.3 straggler 检测](./02-高可用与容错.md) 衔接，但成因在物理层。
- **整机房 / 供电域掉电**：相关性故障（B1）的极端形态；供电冗余、UPS、分阶段恢复。
- **归属**：整机房掉电 = 灾难级 **DR**。
- **参考**：[Power Stabilization for AI Training Datacenters (arXiv 2508.14318)](https://arxiv.org/html/2508.14318v1)、[Wide-Area Power System Oscillations (arXiv 2508.16457)](https://arxiv.org/html/2508.16457v1)、[Oracle GPU Power Smoothing](https://blogs.oracle.com/cloud-infrastructure/behind-the-scenes-gpu-power-smoothing-ai-training)。

## C4. SRE / 运维 / 人因 `[超出代码]`

- **自动化闭环是万卡可用性的真正关键**：检测→归因→隔离坏卡→重调度→续训**全自动**。Meta 466 次故障**只 3 次**需要人工——自动化率直接决定 goodput。
- **坏卡池管理与准入健康检测**：坏卡下线、修复回池前先跑 sniff test / burn-in 才能重新准入（[02 §A.3 sniff test](./02-高可用与容错.md) 是零件，但没讲"坏卡池 lifecycle"这个流程）。
- **变更管理**：固件 / 内核 / 驱动升级本身就是 planned interruption 的大头（Llama 3 中 47/466 是计划内）；灰度、回滚、版本一致性。
- **on-call / runbook / 故障复盘**：把每次故障沉淀成自动化规则。
- **归属**：**支撑层**。

## C5. 安全 / 多租户 / 合规 `[超出代码]`

- checkpoint 与训练数据的**加密与访问控制**（模型权重是高价值资产）。
- 共享集群的**多租户隔离**（网络、存储、调度优先级、抢占）。
- 数据合规与审计（训练数据来源、可追溯）。
- **归属**：**支撑层**（正交）。

## C6. 作业编排与 Pod / 节点级重调度 `[超出代码]`

- **问题**：[02 §5.0 三级重调度阶梯](./02-高可用与容错.md) 里，Megatron+NVRx 能锚到的只到 **L2 进程级**（`ft_launcher` 经 rendezvous 重拉进程）；但进程能被重拉的**前提**是 pod/节点还活着、且被调度器保住。当**整台节点宕机、坏卡需下线换机**时，需要更上一层的 **L3 Pod/节点级重调度**——这层完全在 Megatron 之外，由 **k8s / 集群调度器**承担。
- **为什么万卡才凸显**：万卡训练普遍跑在 k8s 上，节点级故障是常态而非例外；L3 的**重建速度与 gang 语义**直接决定 L2 的 rendezvous 能否成功集结、goodput 多快恢复。
- **[02] 的缺口**：[02 §5](./02-高可用与容错.md) 只到抽象"调度器 / ft_launcher"层，从未展开 pod/k8s 的作业形态与重建流程。补：
  - **训练作业形态**：k8s **Job / JobSet**、**gang scheduling**（要么全部 pod 就绪要么不启动，避免部分 pod 空占卡死等）；operator 如 **Kubeflow Training Operator / Volcano / JobSet controller** 管理一组 pod 的生命周期。
  - **节点级故障处置**：探针发现坏节点 → `cordon` / `drain` 隔离 → 重建 pod 到健康节点 → **坏卡池 lifecycle**（下线、修复、burn-in / sniff test 准入回池——呼应 C4 SRE 与 [02 §A.3 sniff test](./02-高可用与容错.md)、[02 §A.5 打流监控](./02-高可用与容错.md)）。
  - **L3 与 L2 的分工 / 协作**：谁先感知（k8s liveness probe vs FT 段超时，[02 §A.1](./02-高可用与容错.md)）、谁负责拉起（operator 重建 pod vs `ft_launcher` 重拉进程）、rendezvous 如何在**新 pod 上重新集结**——L3 换出健康 pod 后，L2 的 rendezvous（`--nnodes=MIN:MAX`）在新成员集合上重新握手、扩回规模。
  - **拓扑 / gang 约束**：gang scheduling 与拓扑感知放置（呼应 C2）约束了 world size 重组——新 pod 落在哪个机架 / NVLink 域，直接影响 [02 §1.2 弹性 world size](./02-高可用与容错.md) 能否无损重导并行布局，也决定 [02 §5.3 调度补位](./02-高可用与容错.md) 的 cold spare 从哪来、reload 多快（配 [02 §3.2 本地副本 ckpt](./02-高可用与容错.md)）。
- **归属**：集群编排 = **支撑层**；整节点 / 机架换机端触及 **DR**（与 B1 相关性故障衔接）。
- **参考**：[Kubeflow Training Operator](https://www.kubeflow.org/docs/components/trainer/)、[Volcano（gang scheduling / batch）](https://volcano.sh/en/docs/)、[Kubernetes JobSet](https://github.com/kubernetes-sigs/jobset)。
