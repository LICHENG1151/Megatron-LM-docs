## 1. 从"两个问题"到"三个问题"

分析大规模训练的可靠性，一个自然的起点是两个问题：

1. **跑得完** —— 保证训练进程稳定执行到底（≈ [02](./02-高可用与容错.md) 的资源 / 网络 / 调度 / 状态四维）。
2. **算得对** —— 保证结果与中间过程不出计算错误（≈ [02](./02-高可用与容错.md) 的计算正确性：SDC / NaN / spiky loss）。

这两条都是**"正确性 / 存活性"**视角。但在千卡万卡规模，它们之上还缺一个**"效率"**视角——它是前两条的**总目标函数**：

3. **划得来** —— 在故障必然发生的前提下，让 **有效算力占比（goodput / ETTR）** 最大化。

> **为什么必须加第三问**：故障在万卡上不是异常而是**常态**。Meta 训练 Llama 3 405B 用 16384×H100 跑 54 天，**平均每约 3 小时故障一次**（54 天内 466 次中断，其中 419 次意外，78% 硬件相关），却仍做到 **>90% 有效训练时间**。这说明真正的工程目标**不是"消灭故障"**，而是**"让每次故障侵蚀的 goodput 足够小"**。①②都应重述为"goodput 的不同侵蚀源"。

## 2. 三个核心指标：Goodput / ETTR / MFU

| 指标 | 含义 | 衡量什么 | 参考数值 |
|------|------|----------|----------|
| **MFU**（Model FLOPs Utilization） | 实际算力 / 理论峰值算力 | **没故障时**跑得多满 | ByteDance MegaScale 12288 卡训 175B ≈ 55.2% |
| **ETTR**（Effective Training Time Ratio） | 有效训练时间 / 墙钟时间 | **故障 + 恢复 + 存档**吃掉多少（单作业） | Llama 3 > 90% |
| **Goodput** | 集群级聚合有效产出 / 单位时间 | 整个 fleet 的产出效率 | 可归一化到 0~1 的利用率 |

一句话：**MFU 管"跑得满不满"，ETTR 管"被故障与存档吃掉多少"，Goodput 是集群级聚合**。[02](./02-高可用与容错.md) 通篇讲"怎么恢复"，本篇补上"恢复代价如何量化"的目标层。

### 2.1 Goodput 的三级公式

**① 本质定义（比值形式）** —— goodput 就是「有用产出 ÷ 总代价」，写成时间比即 ETTR（单作业口径）：

```
Goodput = T_useful / T_wall = T_useful / (T_useful + T_overhead)
```

- `T_useful`：真正推进收敛的计算时间，**不含**故障后被重算的部分。
- `T_wall`：作业占着资源的墙钟时间。
- 也可用 **token 口径**（Google ML Goodput 常用）：有效 token ÷ 墙钟，**扣掉回滚后重算的 token**。

**② 可算的损失分解（一阶近似，practitioner 常用）** —— 把 overhead 拆成几项独立损失，每项都是「损失时间 / 墙钟」：

```
Goodput ≈ MFU × ( 1 − δ/T − (T/2 + R)/M − f_straggler − … )
                  └───────────────── ETTR ─────────────────┘
```

| 符号 | 含义 | 对应文档 |
|---|---|---|
| `MFU` | 无故障时算力利用率（achieved/peak FLOPs） | 独立乘子 |
| `δ` | 单次 checkpoint **同步写**开销 | [02 §3.5](./02-高可用与容错.md) |
| `T` | checkpoint 间隔 | §4 |
| `M` | MTBF（平均无故障时间） | §3 |
| `R` | 单次故障的重启/恢复时延（= RTO） | [02 §1.1](./02-高可用与容错.md) |
| `T/2` | 每次故障平均重算的进度（回滚到上个 ckpt） | §4 |
| `f_straggler` | 慢节点导致的 barrier 空等占比 | [02 §1.3](./02-高可用与容错.md) |

> 注：`MFU` 与 `f_straggler` **二者取其一，勿重复计入**——若 `MFU` 取**实测值**（已含慢节点拖累，如表上 MegaScale 55.2%），则去掉 `f_straggler` 项；本式按"**健康步理想 MFU + 独立 straggler 项**"分解。

**关键结论**：**Goodput ≈ MFU × ETTR**。MFU 管"没故障时跑多满"，括号里那部分就是 ETTR（被存档 / 故障 / 慢节点吃掉多少）。§4 会对括号项求最优 `T`，得到 Young/Daly 闭式解。

## 3. 故障率随规模近似线性增长

### 3.0 MTBF 是什么（本文各公式里的 `M`）

**MTBF**（Mean Time Between Failures，平均故障间隔时间）= **可修复系统**两次相邻故障之间的平均正常运行时间：

```
MTBF = 总正常运行时间 / 故障次数
```

它就是 §2.1 损失分解与 §4 Young/Daly 公式里的 `M`。直觉上 MTBF 是**故障率 `λ` 的倒数**（`MTBF = 1/λ`）——这正是本节"故障率随规模线性增长 ⇔ MTBF 随规模线性缩短"说的同一件事：16K 卡的 MTBF ≈ 1K 卡的 1/16。

**与几个相邻指标的关系**（容灾/可靠性工程标准词汇）：

| 指标 | 全称 | 含义 | 本文对应 |
|---|---|---|---|
| **MTBF** | Mean Time Between Failures | 平均故障**间隔**（可修复系统） | `M` |
| **MTTF** | Mean Time To Failure | 平均失效前时间（**不可修复**的一次性件，如熔丝） | — |
| **MTTR** | Mean Time To Repair | 平均修复/恢复时间 | ≈ `R`（RTO，见 [02 §1.1](./02-高可用与容错.md)） |

- 一个故障周期 = 正常运行 + 修复，故 **MTBF = MTTF + MTTR**（工程上 MTTR ≪ MTBF 时常近似 MTBF ≈ MTTF）。
- 系统**可用性** `Availability = MTBF / (MTBF + MTTR)`——把本文的 `M` 与 `R` 代入即得，也是 §4 ETTR 里 `R/M` 项的来源。
- **MTBF 越大越可靠**；但万卡级训练里聚合 MTBF 已被规模压到"几十分钟一次"（见下），所以本套文档的重心不是把 MTBF 抬高，而是把 `R`（MTTR）压到秒级、把 `T/2`（RPO）控住——即用 §5 的 HA/DR 两支手段去对冲低 MTBF。

> 注：硬件规格书标的百万小时级 MTBF 是**单卡**值；训练作业关心的是**聚合 MTBF**（数千上万张卡串联，任一张挂即算故障），后者小得多，这正是下面"线性增长"的根因。

### 3.1 规模效应

- 中断率随集群规模近似**线性增长**：16K 卡 ≈ 1K 卡的 **16 倍**中断率。业界数据一致：Microsoft 报每 ~45 分钟一次、ByteDance 每天至少一次、Meta 每 ~3 小时一次。
- **推论**：规模翻倍时，若单次恢复时延不减半，goodput 就下降。这正是 [02 §1.1 in-process restart](./02-高可用与容错.md)（**秒级**原地重组 vs **分钟级**整体重拉）存在的**经济学理由**——[02](./02-高可用与容错.md) 讲了"怎么做到秒级"，本篇点透"为什么规模在逼着你做到秒级"。

## 4. Checkpoint 频率是一个可量化的 goodput 优化问题（Young/Daly）

- 最优存档间隔的一阶近似：`T_opt ≈ √(2 · C · MTBF)`，其中 `C` = 单次存档开销（即 §2.1 的 `δ`），`MTBF` = 平均无故障时间（`M`）。[02 §3.5](./02-高可用与容错.md) 给了"脆弱窗口"的**定性**权衡，本式补上**定量**框架。

### 4.1 从损失分解推出 Young/Daly

对 §2.1 括号里的两项 `δ/T + (T/2)/M` 求最优 `T`（令导数为 0：`δ/T² = 1/(2M)`），即得：

```
T_opt = √(2 · δ · M)          ← Young/Daly
```

代回后，**最优 goodput 有一个简洁的闭式**：

```
ETTR_max ≈ 1 − √(2δ/M) − R/M
```

### 4.2 数值例子（并解释 Meta 为何能到 >90%）

取 `M = 3h = 180 min`：

| 场景 | 参数 | T_opt | ETTR |
|---|---|---|---|
| 同步存档 + 慢重启 | `δ=5, R=5` | `√(2·5·180)≈42 min` | `1 − √(10/180) − 5/180 ≈` **74%** |
| 异步存档 + 秒级重启 | `δ_停顿≈0, R≈1`，每 30 min 异步存档（单次故障丢 ~15 min 重算 + 1 min 重启） | 由 RPO 上限定 | `1 − (15+1)/180 ≈` **91%** |

后者刚好复现 Llama 3 的 >90%——**把 ETTR 从 74% 抬到 91% 的两个杠杆，正是 [02 §1.1 进程内秒级重启](./02-高可用与容错.md)（压低 `R`）与 [02 §3.5 异步 checkpoint](./02-高可用与容错.md)（消掉 `δ/T` 停顿项）**。公式把"为什么要做这两件事"量化了。

### 4.3 RTO / RPO 视角与失效条件

- 用 **RTO / RPO** 这套容灾词汇一套更清晰：
  - **RTO**（Recovery Time Objective，恢复多快）= 检测 + 重启 + reload 时延（即公式里的 `R`）→ **高可用 HA** 优化的量。
  - **RPO**（Recovery Point Objective，能丢多少）= 距上次 checkpoint 的进度（即 `T/2`）→ **checkpoint 间隔**决定，正是 Young/Daly 优化的量 → **容灾 DR** 的命题。
- **重要前提与失效条件**：
  1. Young/Daly 假设故障**无记忆（独立均匀）**。真实训练故障**相关 / 非均匀**（见 [03 §B1 相关性故障](./03-超出框架的工程考量.md)），闭式解只能作参考点。
  2. **异步 checkpoint 让 `δ/T` 停顿项失效**（上表第二行），此时 `T` 不再由 Young/Daly 主导，改由 **RPO 上限**（能容忍丢多少）与**存储带宽**决定；**低 MTBF 时**，"解耦存档频率与存档开销"的异步 / 稀疏 checkpoint 方案往往优于按公式选点——这与 [02 §3.5 异步档](./02-高可用与容错.md) 方向一致。

## 5. 三层归属：高可用 / 容灾 / 效率工程

这套文档已经溢出"高可用"边界，跨进**容灾（DR）**、还有一层 HA/DR 都装不下的**支撑工程**。三者关系是"**一个目标 + 两支手段 + 一层支撑**"：

```
┌──────────────────────────────────────────────────────────────────┐
│  【目标层】 效率 / Goodput（ETTR·MFU）  ← 一切手段的总目标函数       │
└───────────────────────────────┬──────────────────────────────────┘
                 手段服务于目标   │
        ┌────────────────────────┴────────────────────────┐
        ▼                                                  ▼
┌───────────────────────────────┐        ┌───────────────────────────────┐
│  【高可用 HA】优化 RTO(多快恢复)│        │  【容灾 DR】优化 RPO(丢多少)+跨域│
│  局部·频繁·可预期故障           │        │  灾难·故障域级·罕见但致命        │
│  · 资源(in-process restart)    │        │  · Young/Daly 存档间隔 = RPO    │
│  · 网络(超时→abort→重组)       │        │  · 副本 ckpt(跨故障域/异地)     │
│  · 秒级重组·热备               │        │  · 相关性故障→故障域隔离         │
│  · 计算正确性(在线纠错)         │        │  · 存储/异地冷备·整机房掉电      │
│  ⇒ 详见 [02]                   │        │  ⇒ 详见 [02 §3] + [03 §B1/C1/C3]│
└───────────────────────────────┘        └───────────────────────────────┘
        └────────────────────────┬────────────────────────┘
                 两支都依赖        ▼
┌──────────────────────────────────────────────────────────────────┐
│  【支撑层】让 HA/DR 真正生效 —— HA/DR 都装不下的第三类               │
│  可观测性(检测+归因) · 数值正确性/可复现                            │
```

## 6. 三个训练场景特有的"HA/DR 模糊点"

传统 IT（金融/在线服务）里 HA 与 DR 界限分明（HA 保在线、DR 保跨地域），但**大模型训练把这条线打糊了**：

1. **可用性 ≠ 请求成功率，而是 goodput**——训练是**批处理作业**不是在线服务，所以"效率"必须单列为目标层，而非 HA 的一个子项。
2. **checkpoint 同时是 HA 和 DR 的基础原语**——连单卡故障恢复都要回滚 ckpt（DR 动作）。**同一个 ckpt 机制：跨 rank 副本时算 HA，跨机房冷备时算 DR**，很难说某次恢复"纯 HA"还是"纯 DR"。
3. **训练的 DR 很少做跨地域双活**——同步训练扛不住跨域延迟，所以这里的容灾是"**站内跨故障域 + checkpoint 离线冷备**"，而非金融 IT 的"两地三中心 / 双活"。

> **伞形定位**：《训练可靠性工程》= **效率（目标）+ 高可用 + 容灾 + 支撑**。[02](./02-高可用与容错.md) 覆盖左支（HA）+ 计算正确性；[03](./03-超出框架的工程考量.md) 覆盖右支（DR）与支撑层中超出 Megatron 源码的部分。
